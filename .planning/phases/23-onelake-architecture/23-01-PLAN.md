    **Page structure (follow this order exactly):**

    1. **Introduction** (~1 paragraph): Position OneLake as the universal storage substrate of the Enterprise Data Foundation. State the three architect-facing questions the page answers: (a) when to use shortcuts vs mirroring, (b) how the universal namespace eliminates data movement, (c) how security is enforced uniformly across compute engines. State that Data Foundation is not a fifth Citadel layer — it attaches to existing Citadel layers at three points: Layer 1 (APIM), Layer 2 (observability), Layer 4 (Security Fabric).

    2. **The Universal Namespace (LAKE-02)**: Explain OneLake's single namespace — one endpoint for all Fabric compute engines and external tools that speak ADLS. Include the ADLS DFS endpoint URI pattern:
       ```
       abfss://{workspace_id}@onelake.dfs.fabric.microsoft.com/{item_id}.lakehouse/Tables/{table_name}
       ```
       Explain that ADLS Gen2, Dataverse, and SharePoint sources are accessible through this namespace via shortcuts — no data movement, no pipeline, no ETL for open-format sources. Include a `<Note>` that SharePoint/OneDrive shortcuts are in **Public Preview**. Include the ASCII diagram from the research file (the "Agents / Fabric Compute Engines" diagram). State the AI agent implication: agents querying across ADLS, Dataverse, and SharePoint sources see a single namespace and do not need source-specific connectors.

    3. **Shortcut vs Mirroring: The Decision Framework (LAKE-01)**: Open with a statement that the choice determines the security enforcement surface and data movement model — not just ingestion convenience. Then cover all three patterns:

       **Pattern A — Shortcuts (Metadata Pointer, GA)**
       - What: symbolic link; data stays in source; appears as native folder/table in lakehouse
       - When to use: source is open-format (Delta, Parquet, Iceberg, CSV); zero data movement required; source system security must be authoritative; ADLS Gen2 (canonical GA use case), Dataverse, cross-cloud sources
       - Security surface (internal OneLake-to-OneLake shortcuts): passthrough mode — calling user's identity forwarded to target; source system security controls what the user sees; OneLake security roles further filter
       - Security surface (external shortcuts — ADLS from another tenant, S3, GCS, Dataverse): delegated mode — intermediate cloud connection credential; OneLake security roles control what end users see
       - DirectLake exception (include as a `<Note>`): When Power BI semantic models use DirectLake over SQL, or T-SQL engines operate in Delegated identity mode, the item owner's identity is used to access the shortcut target — NOT the calling user's. OneLake security roles still filter results. Source-system policies see the item owner, not the end user. Architects must account for this in governed scenarios.

       **Pattern B — Metadata Mirroring: Catalog Sync, No Data Copy (GA)**
       - What: synchronizes Unity Catalog structure (catalog names, schemas, tables) from Azure Databricks into Fabric; no data moves; OneLake auto-creates shortcuts for underlying Delta/Iceberg data in ADLS
       - When to use: source is Azure Databricks with Unity Catalog; need to expose Databricks-managed Delta tables to Fabric workloads; ADLS-backed Delta files already in place
       - Security surface: SPLIT SURFACE — when Databricks engines read the data, Unity Catalog security (GRANT/REVOKE, row filters, column masks) applies; when Fabric engines (Spark, T-SQL, Power BI) read via the auto-created shortcuts, Fabric workspace security + OneLake security roles apply — Unity Catalog policies do NOT propagate. This must be stated explicitly as a governance boundary architects must design around.

       **Pattern C — Database Mirroring: CDC Replication for Closed-Format Sources (GA for most)**
       - What: CDC replication for sources using closed/proprietary storage (Azure SQL, Cosmos DB, Snowflake, Oracle, SQL Server); data IS replicated into OneLake in open Delta Parquet format
       - When to use: source is an operational database; source uses closed storage format; near real-time analytics without touching the source
       - Security surface: the replicated Delta copy is governed entirely by Fabric workspace roles + OneLake security roles; source system security does NOT propagate to the copy
       - GA status by source: GA — Azure SQL DB, Azure SQL MI, Azure Cosmos DB, Azure Database for PostgreSQL, Snowflake, Oracle, SQL Server; Public Preview — Azure Database for MySQL, Google BigQuery

       **Decision Matrix table**: Create a markdown table with columns SOURCE TYPE | RECOMMENDED PATTERN | SECURITY SURFACE. Include rows for: ADLS Gen2 / Azure Blob (Shortcut external, GA), OneLake / another Fabric item (Shortcut internal, GA), Dataverse (Shortcut external, GA), SharePoint / OneDrive (Shortcut external, Preview), Azure Databricks / Unity Catalog (Metadata mirroring, GA — split surface), Azure SQL DB / Cosmos DB (Database mirroring, GA), Snowflake / Oracle (Database mirroring, GA), Azure DB for MySQL (Database mirroring, Preview).

       **Anti-pattern callout (`<Note>`)**: "Metadata mirroring is zero-copy. Database mirroring is not — it performs CDC replication into OneLake storage. Treating both as zero-copy leads to incorrect security model assumptions."

    4. **Security Enforcement Architecture (LAKE-03)**: Open with a `<Warning>` component: "OneLake security — the feature providing table-, row-, and column-level security roles enforced across all Fabric compute engines — is in Public Preview as of April 2026. Assess production readiness before relying on this feature for enforcement in regulated workloads."

       Then describe the define-once, enforce-everywhere model:
       - Table-level security: controls which tables a principal can see within a lakehouse
       - Row-level security: filters rows based on principal attributes, enforced at read time
       - Column-level security: masks or restricts columns based on principal attributes
       - Enforcement is consistent across Spark, SQL Analytics endpoint, KQL, and Analysis Services — the same role definition applies regardless of engine
       - For AI agents querying via the Databricks SQL REST API (covered in Phase 27/APIM integration): the security enforcement point is Unity Catalog — not OneLake security roles. OneLake security roles apply to Fabric-native compute engines and authorized third-party engines only. State this distinction explicitly.
       - Cross-reference: for the Entra MI chain (agent to APIM to Databricks SQL to OneLake) see the Security and Identity Patterns section.

    5. **GA vs Preview Summary**: Include a brief table summarizing GA/Preview status for all OneLake capabilities mentioned in the page: OneLake shortcuts (GA), Metadata mirroring — Databricks (GA), Database mirroring — Azure SQL/Cosmos DB/Snowflake/Oracle (GA), Database mirroring — MySQL/BigQuery (Preview), OneLake security roles — table/row/column (Preview), SharePoint/OneDrive shortcuts (Preview).

    6. **Connection to Citadel Architecture**: Short section (3-4 sentences) stating that Data Foundation is not a fifth Citadel layer. OneLake attaches at three existing Citadel attachment points: Layer 1 (APIM governs semantic endpoint access to data), Layer 2 (observability tracks data query patterns and latency), Layer 4 (Security Fabric — identity enforcement and column mask propagation). Followed by a `<CardGroup cols={2}>` with two cards: one linking to `architecture/layer-4-security-fabric` ("Layer 4 Security Fabric") and one linking to `architecture/layer-1-governance-hub` ("Layer 1 Governance Hub").

    **Constraints:**
    - No code artifacts — no Python, no Bicep, no YAML config snippets. The ADLS DFS URI is an architecture pattern string, not code.
    - No step-by-step setup instructions. This is an architect-facing design reference, not a how-to guide.

    - SAP Datasphere, multi-region replication, real-time streaming, Power BI Direct Lake setup are out of scope — do not mention.

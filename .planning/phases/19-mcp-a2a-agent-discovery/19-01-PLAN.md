1. **Introduction (2-3 paragraphs)** — EY's AI Agent Development Framework identifies A2A as a foundational protocol for building a connected AI future, with Standard Starter Kit enhancements planned for FY26 Q3. A critical gap in the framework is how agents authenticate and trust one another when crossing organizational or service-line boundaries. Citadel solves this today by using Azure API Management as an A2A trust broker — every inter-agent message passes through a centrally governed gateway where both sender and receiver identities are verified before any payload is delivered.

3. **Cross-Agent Authentication & Federation (section heading + prose + table)** — Explain the three trust patterns APIM enables:

| Trust Pattern | How Citadel Enforces It | EY Relevance |
|---|---|---|
| Same-tenant agents | Both agents authenticate via Entra ID in the same directory; APIM validates tokens against the tenant | Default for agents within the same EY service line |
| Cross-tenant federation | Agent A's tenant trusts Agent B's tenant via B2B or cross-tenant synchronization; APIM validates the federated token | Agents collaborating across EY service lines or with approved external vendors |
| Managed identity to managed identity | Agents use Azure Managed Identities; APIM checks role assignments and access contracts | High-security scenarios where human credentials are not involved |

4. **Security Guarantees for A2A Calls (section heading + prose + bullet list)** — Citadel provides four security guarantees that EY's framework requires but does not yet specify an implementation for:
   - **Non-repudiation**: Every A2A message is logged with sender identity, timestamp, and message hash. Logs are immutable once written to Azure Monitor / Event Hub.
   - **Least-privilege routing**: APIM policies ensure Agent A can only send messages to explicitly allowed recipients. Default-deny applies to all unknown agent pairs.
   - **Payload inspection**: Layer 4 scans A2A message payloads for anomalous patterns, prompt injection, and sensitive data exfiltration before delivery.
   - **Session boundary enforcement**: Long-running A2A sessions must refresh tokens periodically. APIM enforces token expiry and re-authentication.

5. **How This Maps to EY A2A Standards (section heading + prose)** — EY's framework states that future A2A releases will include "Agent Binding i.e. Agent Registration and Discovery." Citadel already provides the trust infrastructure that makes agent binding secure:
   - Before any agent can send or receive A2A messages, it must be registered in Agent 365 and have an access contract in APIM
   - APIM acts as the registration enforcement point — unregistered agents cannot communicate
   - This means EY's future agent registration requirement is already enforceable today through Citadel's Layer 1 + Layer 3 integration

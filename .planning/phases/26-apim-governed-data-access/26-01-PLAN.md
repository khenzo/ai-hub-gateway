    1. **Overview** (~15 lines)
       - Define the governed semantic endpoint pattern
       - Explain how Citadel APIM extends its existing Layer 1 gateway role to govern data access
       - State the dual-purpose of APIM: LLM gateway + semantic endpoint gateway
       - Emphasize: Data Foundation is NOT a fifth layer — it attaches at Layer 1

    2. **The Governed Semantic Endpoint Pattern** (~30 lines)
       - Describe the flow: AI Agent → Citadel APIM → Databricks SQL REST API → Unity Catalog → OneLake
       - Document each hop with specific purpose:
         * Agent: Initiates query via semantic model name (not raw table)
         * APIM: Validates Entra token, applies rate limiting, logs usage
         * Databricks SQL REST API: Executes query against semantic endpoint
         * Unity Catalog: Enforces access controls (row filters, column masks)
         * OneLake: Returns data from underlying Delta tables
       - Include ASCII architecture diagram showing this flow
       - Reference Phase 25 (Semantic Layer) for semantic endpoint concept

    3. **Entra Token Validation** (~20 lines)
       - Document what Entra token validation enforces at the data access boundary:
         * Service Principal / Managed Identity authentication
         * Token scope validation (ensure token is for Databricks resource)
         * Token lifetime and revocation checks
       - Explain why this is Layer 1 (Governance Hub) responsibility
       - Reference existing Layer 1 identity validation documentation

    4. **Usage Tracking and Rate Limiting** (~20 lines)
       - Document APIM-native capabilities applied to data access:
         * Request rate limiting (queries per minute per agent)
         * Token/row consumption tracking for cost attribution
         * Audit logging of all semantic queries
       - Distinguish from LLM token rate limiting: data queries measure result set size, not prompt tokens
       - Explain why data access throttling model differs from LLM throttling

    5. **Dedicated Data Access Policy Fragment** (~30 lines) — PRIMARY SECTION for DSEC-02
       - **ADR-style justification for dedicated fragment:**
         * Risk surface: Data access exposes structured business data; LLM access exposes model APIs — different breach impact
         * Throttling model: Data queries are CPU/storage-bound on Databricks clusters; LLM queries are token-bound on model endpoints — different resource constraints
         * Failure modes: Data access failures require query optimization or cluster scaling; LLM failures require model failover or retry — different operational playbooks
         * Compliance requirements: Data access may require data residency, retention policies, or encryption-at-rest validation that doesn't apply to LLM calls
       - Document the policy fragment structure (XML skeleton):
         * Authentication check (Entra token)
         * Rate limiting (requests per minute)
         * Query logging (audit trail)
         * PII detection (pre-response, links to Phase 27)
       - Explicitly state: "Reusing LLM policy fragments for data access would conflate two distinct risk surfaces and create operational confusion"

    6. **Attachment to the 4-Layer Model** (~15 lines)
       - Clearly state: Data Foundation integration attaches at Layer 1 (Governance Hub / APIM)
       - Show how it reuses existing Layer 1 components:
         * Unified AI Gateway → Extended for semantic endpoints
         * Identity Validation → Entra token validation for data access
         * Smart Operations → Rate limiting adapted for query patterns
       - Reference Layer 1 documentation via CardGroup cross-link
       - State explicitly: "This is NOT a fifth layer"

    7. **Cross-References and Next Steps** (~10 lines)
       - CardGroup linking to:
         * Phase 25: Semantic Layer (prerequisite concept)
         * Phase 27: Security & Identity Patterns (next phase: MI chain and column masks)
         * Layer 1: Governance Hub (existing APIM documentation)
         * ADR-001: APIM Gateway Pattern (foundational decision)

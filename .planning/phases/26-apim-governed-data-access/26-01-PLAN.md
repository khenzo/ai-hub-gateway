3. **ADR-DF-04: AI Agents Route Through APIM, Not Directly to Databricks SQL**
   - Decision Question: Should AI agents call the Databricks SQL REST API semantic endpoint directly, or through Citadel APIM?
   - Why direct calls create a governance gap — name all three concrete risks:
     (a) No Entra token validation at the data layer means any agent with network access can query
     (b) No usage tracking means data access is invisible to cost attribution and audit
     (c) No rate limiting means a runaway agent can exhaust the SQL warehouse compute, affecting all consumers
   - Recommended Answer (opinionated, not neutral): All semantic queries from AI agents route through Citadel APIM. The Databricks SQL REST API warehouse endpoint is registered as an APIM backend. Direct agent access to the SQL warehouse is not permitted in production deployments.

4. **The Governed Semantic Endpoint Pattern** (covers DSEC-01)
   - ASCII architecture diagram showing the full flow. Use the exact diagram from 26-RESEARCH.md:
     AI Agent (spoke VNet) → APIM Gateway (hub VNet) → Databricks SQL Warehouse → Response path back
     Show the policy fragment names inline on the diagram (frag-semantic-entra-auth, rate-limit-by-key, frag-semantic-usage, frag-pii-anonymization)
   - Subsection: Entra Token Validation — what it enforces (audience claim, managed identity, subscription check)
   - Subsection: Usage Tracking — query count not token count; emits to the same Event Hub namespace as LLM usage but with a data-access-specific schema (query_count, warehouse_id fields instead of token_count, model_id)
   - Subsection: Rate Limiting — product-level quota model; queries-per-minute (QPM) per subscription key, not tokens-per-minute (TPM)

5. **Data Access Policy Fragment: Why It Cannot Reuse LLM Fragments** (covers DSEC-02)
   - Framing sentence: Fragment reuse is not just suboptimal — it is architecturally incorrect. Applying frag-llm-usage.xml to SQL REST traffic produces usage records with zero in the token field, making cost attribution and quota enforcement meaningless.
   - Comparison table with ALL THREE structural differences (DSEC-02 requires all three):

   | Dimension | LLM Policy Fragment | Data Access Policy Fragment |
   |-----------|--------------------|-----------------------------|
   | Traffic unit | Tokens (prompt + completion) | Queries (SQL REST calls) |
   | Throttle metric | Tokens per minute (TPM) | Queries per minute (QPM) |
   | Risk surface | Token cost overrun; jailbreak | Data exfiltration; row-count explosion |
   | Failure mode | LLM timeout; content safety block | Warehouse cold start (30–90s spin-up) |
   | Usage schema | token_count, model_id | query_count, warehouse_id |
   | PII handling | Prompt/completion masking | Result column masking |
   | Fragment names | frag-llm-usage.xml | frag-semantic-usage.xml |

   - Brief subsection: SQL Warehouse Cold Start — note as an architectural consideration (not a Pre-GA callout; this is a GA operational property). Serverless SQL warehouses auto-suspend after inactivity; cold start adds 30–90s to the first query. APIM timeout policies for the semantic backend must account for this.

6. **Attachment to the Citadel 4-Layer Model**
   - Explicit: semantic endpoint registration is a Layer 1 extension, not a fifth layer
   - One-paragraph statement of where this sits: APIM is already the Layer 1 runtime enforcement plane. Registering the Databricks SQL warehouse as a new APIM backend extends Layer 1 governance to cover semantic queries. The Data Foundation stack (OneLake → Unity Catalog → Semantic Layer → APIM governed endpoint) connects to the platform at three attachment points: Layer 1 (APIM, this page), Layer 2 (observability, usage telemetry), Layer 4 (Security Fabric, identity — covered in Phase 27).

   ```
   [Developer Workstation]
         |
         v  docker push
   [Azure Container Registry (ACR)]
         |
         v  Helm chart pull / kubectl apply
   [AKS Cluster]
     ├── Namespace: citadel-agents
     │     └── Pod: agent-container (Port 8080)
     │           └── Service (ClusterIP)
     └── Ingress Controller (NGINX/AGIC)
               |
               v  APIM subscription key validation
   [Citadel APIM Gateway — Layer 1]
         |
         v  Entra managed identity check
   [Agent Identity Layer — Layer 3]
         |
         v  traces/evaluations
   [AI Control Plane — Layer 2]
         |
         v  Defender for Containers
   [Security Fabric — Layer 4]
   ```
   Wrap in a code block. Add a brief sentence after explaining the key insight: every agent request enters through APIM (Layer 1), which validates the Citadel access contract before the request reaches the AKS pod.

3. **Prerequisites** — Info callout listing: AKS cluster provisioned, ACR attached to AKS, APIM instance (Citadel Governance Hub), Helm 3 installed, kubectl configured, Azure CLI logged in.

4. **Step 1 — Containerize the Agent** — Steps component with:
   - Create a Dockerfile (Python FastAPI agent example, multi-stage build, non-root user, expose 8080)
   - Build and tag: `docker build -t <acr-name>.azurecr.io/agents/<agent-name>:v1.0.0 .`
   - Push to ACR: `az acr login --name <acr-name> && docker push ...`

5. **Step 2 — Deploy to AKS with Namespace Isolation** — Steps component:
   - Create namespace: `kubectl create namespace citadel-agents`
   - Apply namespace label for Citadel network policy: `kubectl label namespace citadel-agents citadel.io/managed=true`
   - Show minimal Helm `values.yaml` snippet (image.repository, image.tag, service.port, resources.limits)
   - Deploy: `helm upgrade --install <agent-name> ./chart --namespace citadel-agents --values values.yaml`
   - Verify: `kubectl get pods -n citadel-agents`

6. **Step 3 — Wire the Citadel Access Contract** — Steps component:
   - Create APIM API pointing to AKS service URL (internal load balancer or AGIC ingress hostname)
   - Apply subscription key policy in APIM (XML policy snippet — validate-jwt or check-header for `Ocp-Apim-Subscription-Key`)
   - Bind AKS workload identity (managed identity) to APIM backend authentication
   - Test: `curl -H "Ocp-Apim-Subscription-Key: <key>" https://<apim-host>/agents/<agent-name>/health`

7. **Citadel Layer Enforcement Table** — Markdown table with columns: Pipeline Stage | Citadel Layer | Enforcement Mechanism. Rows:
   - Container image push | Layer 4 (Security Fabric) | Defender for Containers image scanning
   - Namespace creation | Layer 3 (Agent Identity) | Entra workload identity binding
   - Ingress routing | Layer 1 (Governance Hub) | APIM API gateway routing
   - Request validation | Layer 1 (Governance Hub) | Subscription key + JWT policy
   - Runtime traces | Layer 2 (AI Control Plane) | Foundry Control Plane agent traces

**Page structure (write these sections in order):**

1. **Intro paragraph** — One short paragraph. Context: the EY AI Agent Development Framework requires all pro-code agents to follow a standardised GitHub repository structure before submission to EYQ Marketplace or deployment to AKS. This page defines the canonical scaffold.

2. **Repository Folder Structure** — Present as a code block (plaintext tree) showing the complete directory layout:
   ```
   <agent-name>/
   ├── .github/
   │   ├── workflows/
   │   │   ├── ci.yml              # Build + test on PR
   │   │   └── cd.yml              # Deploy to AKS on merge to main
   │   └── CODEOWNERS              # EY delivery lead as required reviewer
   ├── chart/                      # Helm chart for AKS deployment
   │   ├── Chart.yaml
   │   ├── values.yaml             # Default values (non-secret)
   │   ├── values.prod.yaml        # Production overrides
   │   └── templates/
   │       ├── deployment.yaml
   │       ├── service.yaml
   │       └── ingress.yaml
   ├── policies/                   # Citadel APIM policy XML files
   │   ├── inbound.xml             # Subscription key + JWT validation
   │   └── outbound.xml            # Response scrubbing policy
   ├── src/                        # Agent source code
   │   ├── agent/
   │   │   ├── __init__.py
   │   │   └── main.py
   │   └── tools/                  # Agent tool implementations
   ├── tests/
   │   ├── unit/
   │   └── integration/
   ├── Dockerfile
   ├── requirements.txt            # or pyproject.toml
   ├── .env.example                # Non-secret env var template
   └── README.md
   ```
   Follow the code block with a brief table: Directory | Purpose. Rows for .github/workflows, chart/, policies/, src/, tests/.

3. **Naming Conventions** — Table: Artefact | Convention | Example. Rows:
   - Repository | `ey-agent-<domain>-<name>` | `ey-agent-finance-invoice-processor`
   - Container image | `<acr>.azurecr.io/agents/<name>:<semver>` | `.../agents/invoice-processor:v1.2.0`
   - Helm release name | `<agent-name>-<env>` | `invoice-processor-prod`
   - Kubernetes namespace | `citadel-agents` (shared) or `citadel-<team>` (team-isolated) | `citadel-finance`
   - APIM API path | `/agents/<name>/v<major>` | `/agents/invoice-processor/v1`

4. **Branch Policy Convention** — Intro sentence: "EY agent repositories follow a three-tier branch strategy to enforce peer review and separation of concerns across dev/test/prod." Then a table: Branch | Purpose | Protection Rules. Rows:
   - `main` | Production-ready code | Require 2 reviewers, require status checks (CI passes), no direct push
   - `develop` | Integration branch for tested features | Require 1 reviewer, require CI pass
   - `feature/<ticket-id>-<description>` | Individual feature or fix work | No restrictions — merges to develop via PR

   After the table, an Info callout: "The `CODEOWNERS` file in `.github/` must name the EY delivery lead as a required reviewer for both `develop` and `main`. This is a mandatory governance control per the EY AI Agent Development Framework (Section 3.4 — EYQ Agent Onboarding Governance)."

5. **Required Repository Secrets** — Table: Secret Name | Purpose | Where to Obtain. Rows:
   - `ACR_LOGIN_SERVER` | Azure Container Registry URL | Azure Portal — ACR Overview
   - `ACR_USERNAME` | ACR service principal or managed identity | Azure Portal — ACR Access Keys
   - `ACR_PASSWORD` | ACR credential | Azure Portal — ACR Access Keys
   - `AKS_KUBECONFIG` | Base64-encoded kubeconfig for AKS | `az aks get-credentials` output
   - `APIM_SUBSCRIPTION_KEY` | Citadel APIM subscription key for integration tests | APIM Portal — Subscriptions

   Add a Warning callout after: "Never commit secrets, `.env` files, or kubeconfig files to the repository. Use GitHub Actions secrets or Azure Key Vault references only."

**Section 2 — ADR-DF-03: Platform-Native vs Tool-Embedded Semantics (H2: ADR-DF-03: Platform-Native vs Tool-Embedded Semantics) [SEML-01]**

H3: Decision Question
State: "Where should business metric definitions live — inside BI tools or inside the data platform?"

H3: The Problem with Tool-Embedded Semantics

Explain: Tool-embedded semantics (DAX in Power BI, LookML in Looker, MDX in SSAS) force organizations to re-implement governance in every tool. When a new BI tool is added, all metric definitions must be re-implemented. When an AI agent needs to query data, it inherits none of this embedded logic — the agent must improvise from raw table schemas.

State the asymmetry directly: the tool-embedded approach creates a "right answer depends on where you ask the question" problem. Different BI tools may calculate ARR Run-Rate differently. An AI agent querying raw tables invents its own calculation. The inconsistency is silent and systemic.

Quote verbatim: "The biggest risk isn't a wrong number. It's a system where the right number depends on where you ask the question."

H3: Modern vs Traditional Comparison

Include the following comparison table verbatim from the research:

| Dimension | Traditional (Tool-Embedded) | Modern (Platform-Native) |
|-----------|----------------------------|--------------------------|
| Location | Inside BI tools (DAX, LookML, MDX) | Inside the data platform, next to the data |
| Governance | Reinvented per tool; fragmented policies | Inherited by construction — row/column policies travel with every metric |
| AI Readiness | Not designed for LLMs; no synonym or guardrail layer | Includes synonyms, explanations, and guardrails; AI agents inherit full governance |
| Reuse | Locked within one tool's proprietary language | SQL + open APIs (REST, JDBC, GraphQL) consumable by any surface |
| Performance | Per-tool caching and aggregates | Shared materialization and query routing across all consumers |
| Versioning | Manual, ad hoc | Semantics as code — CI/CD, Git-versioned, dev → staging → prod |
| Lineage | Rarely visible across tools | Automatic, always-on; impact analysis before any definition change |

H3: Recommended Answer

State the ADR conclusion as an opinionated recommendation (not a neutral comparison):

**Platform-native semantics via MetricFlow + Unity Catalog Business Semantics are required for enterprise AI deployments.**

Quote verbatim: "For enterprise analytics, where consistency and auditability are non-negotiable, the semantic layer is not optional. It is the architecture that makes AI-driven analytics trustworthy."

State the condition that would require re-evaluation: "This recommendation assumes Unity Catalog Business Semantics metric views are defined and certified. If that layer does not exist, text-to-SQL against governed tables with schema documentation is the interim path — with full acknowledgment of its governance limitations."

State when text-to-SQL is acceptable: exploratory analysis, developer prototyping, and human-validated use cases where consistency requirements are relaxed. It is not a production-grade substitute for enterprise AI agents where auditability is required.

**Section 4 — AI Agent Grounding: How Agents Query the Semantic Layer (H2: AI Agent Grounding: How Agents Query the Semantic Layer) [SEML-02]**

Open with the architectural claim: for production AI agents, semantic layer grounding is required, not optional. Text-to-SQL is acceptable only where consistency and auditability requirements are relaxed.

H3: The Two-Phase Agent Interaction

State explicitly that AI agents interact with the semantic layer in exactly two phases:

**Phase 1 — Grounding (before any query is generated)**
The agent reads semantic layer descriptive context: available metrics, dimension definitions, governance rules, synonym mappings, business glossary terms. This prevents hallucinated column names, incorrect joins, and misapplied filters. The agent understands the governed vocabulary before writing any SQL.

**Phase 2 — Execution (governed query, not raw table scan)**
The agent queries governed metric definitions, not raw base tables. Measures, dimensions, filters, and access controls travel with the metric definition. The output is consistent across all consumers because the same metric definition serves every surface.

H3: Why Text-to-SQL Fails at Enterprise Scale

Explain the asymmetry: text-to-SQL generates SQL from schema metadata only. The LLM has no access to business definitions, governance rules, or certified calculation logic. It must infer intent from column names and table relationships. Results are:
- Inconsistent: two agents querying the same question produce different SQL
- Ungoverned: row filters and column masks are not automatically applied
- Unauditable: no lineage from agent query back to certified business logic

H3: Genie as the Semantic Layer Interface

Describe Genie (Databricks AI/BI) as the consumption interface for Unity Catalog Business Semantics: it reads metric definitions, synonyms, and guardrails from the semantic layer before generating any query, and routes execution through governed metric views — not raw table SQL.

State: "When an AI agent queries through the semantic layer, every response is grounded in the same business logic as the BI dashboard and the dbt test. The semantic layer closes the gap between 'what the model produces' and 'what the business defines.'"

**Section 6 — Core vs Edge Semantic Model Governance (H2: Core vs Edge Semantic Model Governance) [SEML-04]**

Open with the governance challenge: enterprise semantic models serve both enterprise-wide reporting requirements (stable, certified, slow-changing) and team-specific, application-specific, and agent-specific requirements (fast-moving, experimental). A flat governance model for both produces either a bureaucratic bottleneck (if enterprise standards apply to everything) or semantic drift (if teams maintain their own separate definitions).

H3: Core: Certified Enterprise Metrics

Define Core:
- Authoritative metric definitions certified at the enterprise level
- Standard dimensions: Customer Segment, Region, Fiscal Period
- Enterprise-wide row-level security and column masking policies
- Changes require formal review and Unity Catalog lineage impact analysis before commit
- Examples: ARR Run-Rate, NRR, Churn Rate, CAC, EBITDA
- Changes slowly — correct rather than fast is the design constraint

H3: Edge: Team-Specific Enrichment

Define Edge:
- Per-team, per-application, or per-agent semantic models seeded from Core
- Inherits all Core definitions and governance (Core metrics available as read-only dependencies)
- Enhanced with local synonyms, domain-specific filters, experimental metrics, agent-specific guardrails
- Team-owned and team-managed — no central review gate for new Edge additions
- Examples: Pipeline Conversion Rate (Sales Edge), FX-Adjusted ARR (Finance Edge), Churn Risk Propensity Score (AI Agent Edge)

H3: The Promotion Path

State the promotion path as an architectural requirement:

"Edge metrics that prove useful and stable must have a defined path to Core. Without a formal promotion path, successful Edge metrics proliferate as parallel definitions — teams reference the Edge version, it becomes the de facto standard, and the Core definition atrophies. The promotion path prevents semantic drift while allowing innovation at the Edge."

Promotion criteria for Edge → Core:
1. Metric has been in production for a defined stabilization period (team-defined)
2. Definition passes Core review against formula standards, dimension alignment, and naming conventions
3. Unity Catalog lineage impact analysis run to identify all Edge dependents before promotion
4. Promoted metric replaces the Edge version — consumers migrate to Core definition with lineage preserved

H3: Governance Enforcement

State: "Core definitions cannot be overridden by Edge consumers. An Edge team can extend a Core metric with local filters and synonyms, but cannot alter the Core formula. This constraint is enforced by Unity Catalog permissions — Core metric views are owned by a governed service account; Edge teams have read-only access to Core definitions."

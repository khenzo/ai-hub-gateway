    2. **ADR: Platform-Native vs Tool-Embedded Semantics (SEML-01)**

       Open with an ADR-style section header: "Architecture Decision: Platform-Native Semantics"

       **The Decision**: The organization commits to platform-native semantic layers (MetricFlow for dbt-native metric definitions + Unity Catalog Business Semantics for Databricks-native metric governance) as the single source of metric truth. Tool-embedded semantics (Power BI measures, Tableau calculated fields, Looker explores, Excel formulas) are explicitly designated as presentation-layer only — they may consume platform-native metrics but cannot define new canonical metrics.

       **Why This Is Non-Negotiable for AI Agents** — enumerate the risks of tool-embedded semantics for AI agent grounding:
       - **Invisibility**: AI agents cannot see Power BI measures or Tableau calculations. When an agent queries "what is our Q3 revenue?", it has no access to the business logic encoded in a Power BI DAX measure. It falls back to raw table columns and must reconstruct the logic — incorrectly.
       - **Inconsistency**: Two BI tools may define "active customer" differently. An agent querying each tool gets different answers. Without a platform-native canonical definition, there is no ground truth to reconcile them.
       - **Tool Lock-in**: Metrics trapped in Tableau cannot be consumed by a Python notebook, an AI agent, or a new BI tool. Platform-native metrics are tool-agnostic by design.
       - **Governance Gap**: Tool-embedded metrics bypass Unity Catalog governance. Row filters and column masks apply at the table level, but a tool-embedded calculation may aggregate across rows that should be filtered, or expose columns that should be masked.
       - **Version Drift**: When the underlying table schema changes, tool-embedded measures break silently. Platform-native metrics are validated against the semantic layer schema at query time.

       **The Complementary Layers Pattern**: Explain that MetricFlow and Unity Catalog Business Semantics are NOT competitors — they are complementary:
       - **MetricFlow** (dbt): The authoring layer. Data engineers define metrics in dbt YAML using the dbt Semantic Layer. MetricFlow handles metric logic (aggregations, filters, dimensions, derived metrics). Best for: complex metric definitions, dbt-native pipelines, multi-environment promotion.
       - **Unity Catalog Business Semantics** (Databricks): The governance layer. Metrics are registered in Unity Catalog alongside tables, with the same GRANT/REVOKE, row filter, and column mask enforcement. Best for: Databricks-native consumption, workspace-level access control, integration with Databricks AI/BI and SQL analytics.
       - **Integration point**: MetricFlow metrics can be materialized as tables/views in Unity Catalog; Unity Catalog can expose those tables as governed semantic objects. The metric definition lives in dbt YAML (authoring); the governance envelope lives in Unity Catalog (access control).

       Include a `<Note>` callout: "MetricFlow is GA on dbt Cloud and dbt Core 1.6+. Unity Catalog Business Semantics (metrics and dimensions registered in Unity Catalog) should be verified for GA status on your Databricks runtime before production adoption. If Business Semantics is in Public Preview, wrap this section in a `<Warning>` callout."

       **When Tool-Embedded Semantics Are Acceptable**: A brief subsection listing the only acceptable uses: presentation formatting (currency symbols, date formatting), tool-specific visual calculations (running totals in a chart), and temporary exploratory analysis. All of these must consume — not redefine — platform-native metrics.

    3. **AI Agents and the Semantic Layer (SEML-02)**

       Open with the distinction: "Grounding vs Text-to-SQL"

       **Why Text-to-SQL Is Insufficient for Production Agents**:
       - Text-to-SQL generates raw SQL against tables. It requires the agent to understand table schemas, join paths, aggregation logic, filter conditions, and business definitions — all from natural language context.
       - Production risk: an agent asked "show me revenue by region" might join the wrong tables, use the wrong date column (created_at vs order_date), apply the wrong aggregation (SUM vs COUNT DISTINCT), or miss a critical filter (exclude cancelled orders).
       - Governance risk: text-to-SQL bypasses the semantic layer entirely. The agent queries raw tables directly, missing row filters and column masks enforced at the semantic layer.

       **How Agents Query the Semantic Layer**:
       - The agent sends a natural language query to the semantic layer API (e.g., dbt Semantic Layer API, Databricks Genie/AI/BI API, or a custom semantic endpoint fronted by APIM).
       - The semantic layer resolves the query to a pre-defined metric: "revenue" = `SUM(order_total) WHERE status != 'cancelled' AND order_date BETWEEN {start} AND {end}`.
       - The semantic layer handles joins, filters, aggregations, and time-grain automatically. The agent does not write SQL.
       - The result is governed: Unity Catalog row filters and column masks apply at query time. The agent receives only data the user is authorized to see.
       - The answer is consistent: the same metric definition is used regardless of which agent, dashboard, or notebook asks the question.

       Include an ASCII diagram:
       ```
       AI AGENT (Natural Language)

            ▼
       SEMANTIC LAYER API (APIM-governed endpoint)

Text-to-SQL is acceptable for prototyping and internal tooling. It is not acceptable for customer-facing or compliance-bound agent workloads."

    4. **Author-Once-Reuse-Everywhere: The Metric Lifecycle (SEML-03)**

       Trace a single metric definition through its entire lifecycle. Use a concrete example: `revenue` (SUM of order_total, excluding cancelled orders, in the local currency).

       **Step 1 — Authorship in dbt YAML (MetricFlow)**:
       - Data engineer defines the metric in `models/marts/revenue.yml`:
         ```yaml
         metrics:
           - name: revenue
             description: "Total revenue from completed orders, excluding cancellations"
             type: simple
             type_params:
               measure: order_total
             filter: |
               {{ Dimension('order__status') }} != 'cancelled'
         ```
       - The metric is version-controlled, code-reviewed, and tested in CI/CD (dbt build + dbt test).
       - `persist_docs` makes the description visible in Unity Catalog Business Glossary (link to Phase 25 dbt page if it exists).

       **Step 2 — Materialization in Unity Catalog**:
       - dbt run materializes the underlying model (`analytics.gold.orders`) in Unity Catalog.
       - The metric definition remains in MetricFlow metadata — it is not "baked into" the table. The table contains rows; the metric contains logic.
       - Unity Catalog registers the metric as a semantic object with the same namespace (`analytics.gold.revenue`) and applies GRANT, row filters, and column masks.

       **Step 3 — Consumption by Dashboards**:
       - Power BI connects to the Databricks SQL endpoint and queries `analytics.gold.revenue` via DirectQuery.
       - The semantic layer resolves the metric logic. Power BI receives aggregated results, not raw rows.
       - The metric definition in Power BI is a pass-through reference — not a redefinition. The DAX measure is: `Revenue = SELECTEDVALUE(analytics.gold.revenue)`.

       **Step 4 — Consumption by Notebooks**:
       - Data scientist opens a Databricks notebook and queries the semantic layer API: `sl.query(metrics=["revenue"], group_by=["region"], where="order_date BETWEEN '2026-07-01' AND '2026-09-30'")`.
       - The same metric logic applies. The data scientist does not write SQL.

       **Step 5 — Consumption by AI Agents**:
       - AI agent receives natural language: "What was our Q3 revenue?"
       - Agent calls the semantic layer API (fronted by APIM — Phase 26 covers this integration).
       - The semantic layer resolves to the same `revenue` metric, applies the same filters, and returns the same result.
       - The agent's response is grounded in the canonical metric — not a text-to-SQL guess.

    5. **Core vs Edge Semantic Model (SEML-04)**

       Open with the governance challenge: "As the number of metrics grows, a single flat namespace becomes unmanageable. Some metrics require enterprise-level certification; others are team-specific experiments. The Core vs Edge model provides a governed promotion path."

       **Core Metrics (Enterprise-Certified)**:
       - Definition: Metrics that are authoritative, broadly reusable, and governed by the central data platform team.
       - Characteristics: defined in the `core` catalog (e.g., `core.finance.revenue`, `core.operations.active_users`), rigorously tested, documented with business context, subject to change-control process, mapped to business glossary terms, reviewed by data stewards.
       - Governance: Unity catalog GRANT at catalog level — all consumers have access; row filters and column masks apply uniformly; lineage is complete and audited; dbt tests enforce data quality (uniqueness, not null, referential integrity).
       - Ownership: Central data platform team or dedicated data stewards.

       **Edge Metrics (Team-Specific)**:
       - Definition: Metrics created by individual teams for domain-specific analysis. They are not enterprise-certified and may change without central approval.
       - Characteristics: defined in team-specific catalogs (e.g., `marketing.edge.campaign_roi`, `sales.edge.pipeline_velocity`), created by domain engineers, documented lightly, subject to team-level governance only, may reference Core metrics as building blocks.
       - Governance: Unity catalog GRANT restricted to team workspace; row filters may be team-specific; no central data steward review required; dbt tests are team responsibility.
       - Ownership: Domain team that created the metric.

       **The Promotion Path: Edge → Core**:
       A governed process for promoting a metric from Edge to Core:
       1. **Request**: Team submits a promotion request with business justification, metric definition, and intended consumers.
       2. **Review**: Central data steward reviews the metric for: alignment with existing Core metrics (no duplicates, no conflicting definitions), data quality (dbt test coverage, lineage completeness), documentation (business description, ownership, refresh schedule), governance (GRANT model, row filter/column mask requirements).
       3. **Certification**: Metric is copied/refactored into the `core` catalog namespace (e.g., from `marketing.edge.campaign_roi` to `core.marketing.campaign_roi`). The Edge version is deprecated with a deprecation notice.
       4. **Announcement**: Metric is published in the Business Glossary and announced to all consumers.
       5. **Ongoing**: Core metric is subject to standard change control. Updates require steward approval.

       Include an ASCII diagram:
       ```
       EDGE (Team-Specific)
       │
       ├─ marketing.edge.campaign_roi
       ├─ sales.edge.pipeline_velocity
       └─ product.edge.feature_adoption
       │
       │  Promotion Path (governed review)
       │  ├── Business justification
       │  ├── Data quality validation
       │  ├── Documentation review
       │  └── Steward approval
       ▼
       CORE (Enterprise-Certified)
       │
       ├─ core.finance.revenue
       ├─ core.operations.active_users
       └─ core.marketing.campaign_roi  ← promoted from Edge
       │
       └── Consumed by: dashboards, notebooks, AI agents
       ```

       **Anti-patterns** (`<Note>` callout):
       - **Shadow Core**: Teams creating Edge metrics that duplicate Core metrics because the Core metric is "too slow to update" or "doesn't have my filter". This indicates a governance failure, not a metric gap.
       - **Premature Core**: Promoting an Edge metric to Core before it has proven value. Core metrics carry governance overhead — only promote metrics with broad, sustained demand.
       - **Tool-Embedded Core**: Defining a Core metric in a BI tool and then trying to reverse-engineer it into the platform-native layer. Always author platform-native first, then consume in tools.

       - **Layer 4 (Security Fabric)**: Unity Catalog column masks and row filters propagate through the semantic layer to all consumers. Entra Managed Identity governs the credential chain. (Phase 27 covers the security identity chain design.)

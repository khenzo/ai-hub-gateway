  <name>Create ADR: APIM Gateway Pattern</name>
  <files>docs/architecture/decisions/adr-001-apim-gateway-pattern.mdx</files>
  <action>
    Create Architecture Decision Record for APIM Gateway Pattern covering:
    
    **Context:**
    - Need for unified AI gateway to govern all AI traffic
    - Requirements: LLM governance, AI resiliency, cost attribution, policy enforcement
    - Enterprise need for centralized control and observability
    
    **Decision:**
    - Use Azure API Management as the unified AI gateway
    
    **Alternatives Considered:**
    - Direct model access (rejected: no governance)
    - Custom gateway solution (rejected: maintenance burden)
    - Application Gateway + WAF (rejected: not AI-specific)
    - Azure Front Door (rejected: different use case)
    
    **Rationale:**
    - Built-in AI gateway capabilities (token rate limiting, semantic caching)
    - LLM routing and load balancing
    - Policy-as-code enforcement
    - Native integration with Azure ecosystem
    - Enterprise-grade security and observability
    - Multi-cloud/hybrid support
    
    **Consequences:**
    - Positive: Unified control plane, proven patterns, reduced custom code
    - Positive: Built-in AI resiliency features
    - Trade-off: APIM learning curve for teams
    - Trade-off: Additional cost for APIM service
    
    **Related Decisions:**
    - ADR-002: Hub-spoke networking
    - ADR-003: Landing Zone integration
  </action>

<task type="auto">
  <name>Create ADR: Hub-Spoke Networking</name>
  <files>docs/architecture/decisions/adr-002-hub-spoke-networking.mdx</files>
  <action>
    Create Architecture Decision Record for hub-spoke networking:
    
    **Context:**
    - Need for centralized governance with distributed agent workloads
    - Requirement to support multiple business units/use cases
    - Need for network isolation while maintaining connectivity
    - Alignment with Azure Landing Zone architecture
    
    **Decision:**
    - Implement hub-spoke topology with CGH as hub and CAS as spokes
    
    **Alternatives Considered:**
    - Flat network (rejected: no isolation, security concerns)
    - Mesh topology (rejected: complexity, routing overhead)
    - Single VNet (rejected: doesn't scale, blast radius)
    - Fully isolated (rejected: no collaboration, duplicate infrastructure)
    
    **Rationale:**
    - Separation of concerns with unified oversight
    - Local freedom in spokes under central governance
    - Proven Azure Landing Zone pattern
    - Scales from 10 to 10,000 agents
    - Network isolation per business unit
    - Centralized security enforcement at hub
    - Cost attribution per spoke
    
    **Consequences:**
    - Positive: Clear governance boundary, scalable, secure by default
    - Positive: Aligns with enterprise ALZ investments
    - Trade-off: VNet peering complexity
    - Trade-off: Network latency between hub and spokes
    
    **Implementation Notes:**
    - Two deployment patterns supported (hub network vs spoke network)
    - VNet peering with gateway transit
    - Private endpoints for all services
  </action>
  <verify>ADR explains hub-spoke choice with clear rationale</verify>
  <done>ADR-002 Hub-Spoke Networking created</done>
</task>

<task type="auto">
  <name>Create ADR: Landing Zone Integration</name>
  <files>docs/architecture/decisions/adr-003-landing-zone-integration.mdx</files>
  <action>
    Create Architecture Decision Record for Azure Landing Zone integration:
    
    **Context:**
    - Citadel is designed to integrate with existing Azure environments
    - Organizations have existing ALZ investments
    - Need to leverage shared services (identity, networking, monitoring)
    - AI workloads are application landing zones
    
    **Decision:**
    - Design Citadel as supplemental AI landing zone that extends ALZ
    
    **Alternatives Considered:**
    - Standalone deployment (rejected: duplicate infrastructure, security gaps)
    - Replace ALZ (rejected: too disruptive, not feasible)
    - Fully separate AI environment (rejected: operational complexity)
    
    **Rationale:**
    - Reuses existing ALZ investments (hub VNet, DNS, monitoring)
    - Follows CAF guidance for application landing zones
    - Consistent with ALZ governance model
    - Accelerates deployment via shared infrastructure
    - Maintains enterprise security standards
    - Enables gradual adoption
    
    **Consequences:**
    - Positive: Faster time to value, leverages existing governance
    - Positive: Consistent with enterprise architecture
    - Trade-off: Dependency on ALZ implementation quality
    - Trade-off: Requires coordination with platform teams
    
    **Integration Points:**
    - Shared hub VNet and network infrastructure
    - Centralized Log Analytics and monitoring
    - Enterprise Entra ID for identity
    - Azure Policy for governance
    - Shared DNS infrastructure
  </action>
  <verify>ADR explains ALZ integration approach with clear rationale</verify>
  <done>ADR-003 Landing Zone Integration created</done>

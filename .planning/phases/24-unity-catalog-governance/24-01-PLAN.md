**Page structure (six sections in this order):**

**Section 1 — Introduction (H2: Unity Catalog as the Governance Backbone)**
Central thesis: Unity Catalog achieves compute/governance separation because all governance metadata — privileges, row filters, column masks, tags, lineage — is stored in the metastore, a control-plane service managed by Databricks that is entirely independent of cluster lifecycle. When a cluster terminates, governance is unaffected. When a dbt model drops and recreates a table, governance persists.

State that this page covers: compute/governance separation, ABAC design for scale, the three-level namespace as the downstream consumer contract, and the Iceberg REST + OneLake integration path.

Add a Note callout: "Data Foundation connects to the existing SOF1A 2.0 / Citadel 4-layer model at three attachment points — Layer 1 (APIM governance), Layer 2 (observability), and Layer 4 (Security Fabric). Unity Catalog is the governance substrate within that model, not an additional layer."

**Section 2 — Compute vs Governance Separation (H2: Compute vs Governance Separation) [UCAT-01]**

Sub-sections:
- H3: The Metastore as the Control Plane — explain what "separated" means: cluster = data plane; metastore = control plane. All governance objects (GRANT, REVOKE, row filters, column masks, tags, lineage) are stored in the metastore control-plane service. They are evaluated at query time regardless of which cluster or SQL warehouse executes the query.
- H3: What Persists Across the Cluster Lifecycle — bullet list of what survives cluster termination AND table rebuilds: schema-level GRANT (applies to current AND future tables in the schema — no re-grant after dbt rebuild), row filters, column masks (evaluated at query time by metastore), tags, lineage. Explicitly note: "When a dbt incremental run drops and recreates a table, the new table inherits the schema-level GRANT automatically. No re-grant is required."
- H3: Lineage as a Permanent Audit Trail — describe lineage retention: captured at runtime across all supported compute interfaces; retained indefinitely (all lineage from September 1, 2024 onward); accessible across all workspaces attached to the same metastore. Architectural implication: an AI agent's access to a certified metric is permanently auditable through lineage system tables, independent of which cluster processed the query.

**Section 3 — ABAC Design Pattern (H2: ABAC Design Pattern for Scale) [UCAT-02]**

Open immediately with a Warning callout:
```
<Warning>
Unity Catalog Attribute-Based Access Control (ABAC) is in **Public Preview** as of March 2026. It requires Databricks Runtime 16.4 or above, or serverless compute, to enforce policies. Users not subject to an ABAC policy can use any runtime. Validate ABAC on your target runtime before adopting it as your primary governance model for production workloads.
</Warning>
```

Sub-sections:
- H3: Why ABAC Over RBAC at Scale — contrast: RBAC requires `GRANT SELECT ON TABLE` for every new table created; ABAC policies attach at catalog or schema level and apply automatically to all current and future tables in scope. New teams onboard without any per-table grant.
- H3: Tags as Governance Attributes — what can be tagged: catalogs, schemas, tables, columns, views, volumes, models. Governed tags. SQL syntax note (SET TAG on DBR 16.1+; SET TAGS on DBR 13.3+). Constraints: 50 tags max per object.
- H3: Tag Inheritance for ABAC Policy Evaluation — CRITICAL accuracy: tags applied at catalog or schema level are **implicitly inherited** by child objects **when Unity Catalog evaluates ABAC policies**. This inheritance does NOT apply to general tag queries or INFORMATION_SCHEMA.TABLE_TAGS. Always qualify this as "for ABAC policy evaluation only."
- H3: How New Teams Onboard Without Per-Table ACL Updates — step-by-step walkthrough using the pattern from research: new schema added, admin tags the schema with `domain=team_a`, existing catalog-level ABAC policy applies automatically to all tables in that schema. No table-level policy update.
- H3: Policy Types and the Single-Filter Constraint — two ABAC policy types: row filter policies, column mask policies. Architecture constraint: only one distinct row filter can resolve at runtime for a given user and table; if multiple distinct row filters apply, Databricks blocks access and throws an error. Design recommendation: define policies at the highest applicable level (catalog) to minimize filter collision risk.
- Include the ABAC ASCII diagram from research:

```
CATALOG: analytics   [tag: env=prod]
  │
  ├─ SCHEMA: silver  [tag: domain=finance]
  │    │
  │    ├─ TABLE: transactions   (inherits both tags for ABAC evaluation)
  │    └─ TABLE: invoices       (inherits both tags for ABAC evaluation)
  │
  └─ SCHEMA: gold    [tag: domain=executive]
       │
       └─ TABLE: executive_metrics  (inherits env=prod, domain=executive)

ABAC POLICY (defined once at catalog level):
  IF tag domain=finance THEN apply row_filter(region = user_region)
  IF tag domain=executive THEN apply column_mask(salary → [REDACTED] unless exec_role)

New table added to silver schema → automatically governed.
New schema added to analytics catalog → automatically governed at catalog level.
No per-table GRANT or policy update required.
```

- Note that row filters and column masks applied directly per table (`ALTER TABLE SET ROW FILTER`) are GA as of DBR 12.2+ — the Public Preview status applies to ABAC-managed application of row filters and column masks only.

**Section 4 — Three-Level Namespace as Governance Contract (H2: The Three-Level Namespace as Governance Contract) [UCAT-03]**

Sub-sections:
- H3: catalog.schema.table as the Stable Consumer Surface — explain: all governance operations (GRANT, REVOKE, row filter, column mask, tag, lineage) use the logical namespace. Physical storage (ADLS path) is an implementation detail managed internally by Unity Catalog. Storage path can change; the governance contract does not.
- H3: What Downstream Consumers Depend On — when a downstream consumer holds a reference to `analytics.gold.customer_metrics`, that reference remains valid regardless of: which cluster last wrote to the table, whether the ADLS storage account was reorganized, whether the table was rebuilt (CTAS, dbt rebuild, CLONE), or whether the Databricks workspace version was upgraded. The consumer depends on the governance contract, not a physical file path.
- H3: Workspace-Catalog Binding for Environment Isolation — Unity Catalog allows binding specific catalogs to specific workspaces. Provides dev/staging/prod environment isolation without changing the namespace. A downstream consumer with access only to the `prod` workspace sees only catalogs bound to that workspace. The namespace is unchanged; binding controls visibility, not naming.
- H3: The dbt Integration Guarantee — make explicit: "Schema-level grants are the mechanism by which Unity Catalog separates the data governance lifecycle from the data pipeline lifecycle. When dbt drops and recreates a table, no re-grant is required."

**Section 6 — Connection to the Citadel Architecture (H2: Unity Catalog in the Citadel Architecture)**

State the three attachment points explicitly:
- **Layer 1 (APIM / Governance Hub):** APIM governs semantic endpoint access to Databricks SQL REST API — Unity Catalog enforces row filters and column masks on the data returned through that endpoint. (Phase 27 covers the APIM integration design.)

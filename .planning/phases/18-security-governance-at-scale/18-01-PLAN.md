**Page sections (write full MDX, not just headings):**

1. **Introduction (2-3 paragraphs)** — EY's AIRA and AI QRM processes require evidence that AI systems are registered, risk-assessed, and subject to ongoing monitoring. Citadel integrates three Azure services — Azure Policy, Defender for AI, and Microsoft Purview — that generate the compliance evidence and automated enforcement signals needed to support these processes. This page explains which service maps to which EY requirement.

2. **Overview mapping table** — High-level mapping of each Azure service to EY compliance requirement:

| Azure Service | EY Compliance Requirement | Coverage | Enforcement Type |
|---|---|---|---|
| Azure Policy initiatives | AI system configuration standards | Automated | Enforces baseline configuration rules across AI resources; violations block deployment |
| Defender for AI | InfoSec review — threat signal evidence | Partial | Provides real-time threat signals (jailbreak attempts, prompt injection); human review still required |
| Microsoft Purview | Data Protection review — data classification evidence | Partial | Automates data classification and PII scanning; formal DP approval remains manual |

3. **Azure Policy — Configuration Governance (section heading + content)** — Explain how Azure Policy initiatives apply to Citadel. Include:
   - What Azure Policy enforces: APIM SKU compliance, private endpoint requirements, network isolation rules, diagnostic settings enablement
   - How it maps to firmwide governance: Any Citadel deployment that violates a policy initiative is flagged and can be blocked at the resource group level
   - Note on AIRA evidence: Azure Policy compliance reports can be exported as evidence of configuration controls for AIRA submissions
   - Example policy categories table:

| Policy Initiative | Controls Enforced | Maps to EY Requirement |
|---|---|---|
| AI Services Governance | Require private endpoints; disable public network access | InfoSec — network isolation |
| API Management Governance | Enforce APIM Standard v2+; require diagnostic settings | InfoSec — audit logging enablement |
| Data Residency | Restrict AI resources to approved Azure regions | Data Protection — data residency |
| Identity Controls | Enforce Managed Identity for AI resource access | InfoSec — identity governance |

4. **Defender for AI — Threat Signal Collection (section heading + content)** — Explain Defender for AI's role:
   - What it detects: Jailbreak attempts, prompt injection, anomalous token usage, model abuse patterns
   - How threat signals feed into AIRA/AI QRM: Defender for AI generates security alerts that teams include as evidence in AIRA risk classification and AI QRM reviews. High-severity signals may trigger re-review.
   - What Citadel's Layer 4 adds: Layer 4 (Security Fabric) acts as the real-time enforcement layer; Defender for AI provides post-hoc threat intelligence and pattern analysis at the subscription level
   - Automated vs Manual boundary table:

| Defender Signal | Citadel Layer 4 Action | EY Process Impact |
|---|---|---|
| Jailbreak attempt detected | Layer 4 blocks request in real-time | Alert stored for AIRA review evidence |
| Prompt injection detected | Layer 4 blocks request in real-time | Alert included in InfoSec review |
| Anomalous token usage | Alert generated | Manual review triggered for AI QRM |
| Model exfiltration attempt | Alert generated | InfoSec escalation required |

5. **Microsoft Purview — Data Classification (section heading + content)** — Explain Purview's role:
   - What Purview provides: Automated scanning and classification of data assets flowing through AI workloads, sensitivity labeling, PII detection
   - How it supports Data Protection review: Purview generates a classification report that serves as evidence for the DP review required in AIRA. The formal review and approval remain manual.
   - How it integrates with Layer 4: Layer 4 PII blocking is rule-based (regex + Language service); Purview adds enterprise-wide classification that can inform which PII patterns to configure in APIM policies
   - Classification-to-action table:

| Purview Classification | Layer 4 Response | EY Process Step |
|---|---|---|
| PII — High Sensitivity | PII blocking policy active (block or anonymize) | Triggers Data Protection review in AIRA |
| Confidential | Audit logging at enhanced verbosity | Noted in AIRA risk classification |
| General | Standard logging | No additional AIRA steps required |

6. **Automated vs Manual Control Summary** — A clear visual summary of the enforcement boundary (call this out with a `<Note>` component):
```
<Note>
**Citadel automates the enforcement runtime. EY AIRA and AI QRM remain manual processes.**

The following controls require human action outside Citadel:
- Registering AI systems in EY's AIRA application
- Completing AI QRM reviews for medium/high-risk agents
- Obtaining formal Data Protection and InfoSec approvals
- Conducting service-line-specific reviews (e.g., TTAR AI Agent Acceptance Process)

Citadel's role is to generate the compliance evidence (audit logs, threat signals, classification reports, policy compliance reports) that these manual processes require.
</Note>
```

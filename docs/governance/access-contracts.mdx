# 📋 AI Access Contracts

## Introduction

**AI Access Contracts** are policy bundles that govern how applications, agents, and users interact with AI services. Unlike traditional API keys that only provide access, contracts encode complete governance policies including rate limits, content safety rules, cost controls, and audit requirements.

## What Are AI Access Contracts?

### Definition and Purpose

An AI Access Contract is a declarative document that binds:

- **Identity**: Who is accessing (user, application, agent)
- **Policies**: What rules apply (rate limits, safety, cost)
- **Resources**: Which AI services can be accessed
- **Conditions**: When and where access is permitted

### Contract vs. Traditional API Keys

| Aspect | API Key | AI Access Contract |
|--------|---------|-------------------|
| **Scope** | Binary (access/no access) | Granular policies |
| **Rate Limits** | Global or none | Per-contract quotas |
| **Safety Controls** | External/optional | Built-in enforcement |
| **Cost Attribution** | Difficult | Built-in tracking |
| **Audit Trail** | Basic logging | Comprehensive telemetry |
| **Lifecycle** | Manual rotation | Automated with expiry |

### Benefits for Enterprise Governance

1. **Policy Consistency**: Same contract applied across environments
2. **Self-Service**: Developers request access through contracts
3. **Cost Control**: Per-contract budgets prevent overruns
4. **Compliance**: Built-in audit trails and policy enforcement
5. **Flexibility**: Different contracts for different use cases

| Field | Description | Example |
|-------|-------------|---------|

## Contract Lifecycle

### Creation and Approval Workflow

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   Request    │ →  │   Review     │ →  │   Approve    │ →  │   Create     │
│              │    │              │    │              │    │              │
│ Developer    │    │ Platform     │    │ Platform     │    │ Contract     │
│ submits      │    │ Team reviews │    │ Lead         │    │ generated    │
│ contract     │    │ policies     │    │ approves     │    │ with ID      │
│ request      │    │              │    │              │    │              │
└──────────────┘    └──────────────┘    └──────────────┘    └──────┬───────┘
                                                                   │
                                                                   ▼
                                                          ┌──────────────┐
                                                          │   Deploy     │
                                                          │   to APIM    │
                                                          └──────────────┘
```

### Deployment to APIM

Contracts are deployed as APIM products with associated policies:

```xml
<!-- APIM Policy enforcing contract terms -->
<policies>
  <inbound>
    <!-- Rate limiting from contract -->
    <rate-limit calls="1000" renewal-period="3600" />
    
    <!-- Token quota from contract -->
    <quota calls="10000" renewal-period="86400" />
    
    <!-- Content safety check -->
    <send-request mode="new" response-variable-name="safety-check">
      <set-url>{{content-safety-endpoint}}/text:analyze</set-url>
      <set-method>POST</set-method>
      <set-header name="Ocp-Apim-Subscription-Key">
        <value>{{content-safety-key}}</value>
      </set-header>
    </send-request>
    
    <!-- Cost header injection for tracking -->
    <set-header name="x-contract-id">
      <value>{{contract-id}}</value>
    </set-header>
  </inbound>
</policies>
```

### Activation and Monitoring

Once deployed:

1. **Activation**: Contract status changes to `active`
2. **Key Distribution**: API keys or credentials provided to requester
3. **Monitoring**: Usage tracked against contract quotas
4. **Alerting**: Notifications at threshold breaches

### Updates and Versioning

Contract changes follow semantic versioning:

| Change Type | Version Impact | Example |
|-------------|----------------|---------|
| **Patch** | Increase patch (1.0.0 → 1.0.1) | Description update |
| **Minor** | Increase minor (1.0.0 → 1.1.0) | New optional policy |
| **Major** | Increase major (1.0.0 → 2.0.0) | Breaking policy change |

### Retirement and Revocation

```
Contract Retirement Process:

1. Deprecation Notice (30 days before)
   ↓
2. Grace Period (existing requests honored)
   ↓
3. Suspension (no new requests)
   ↓
4. Revocation (all access blocked)
   ↓
5. Archive (logs retained, contract deleted)
```

## Contract Enforcement Points

### APIM Policy Configuration

Contracts map to APIM products with associated policies:

```xml
<policies>
  <inbound>
    <!-- Base policy: Authentication -->
    <base />
    
    <!-- Contract-specific rate limits -->
    <rate-limit-by-key 
      calls="{{contract.requestsPerHour}}"
      renewal-period="3600"
      counter-key="{{contract.id}}" />
    
    <!-- Token quota management -->
    <quota-by-key 
      calls="{{contract.tokensPerDay}}"
      renewal-period="86400"
      counter-key="{{contract.id}}" />
    
    <!-- Content safety integration -->
    <choose>
      <when condition="{{contract.contentSafetyEnabled}}">
        <send-request mode="new" response-variable-name="safety-result">
          <set-url>{{content-safety-endpoint}}/text:analyze</set-url>
          <set-method>POST</set-method>
        </send-request>
        <choose>
          <when condition="@(((IResponse)context.Variables["safety-result"]).StatusCode != 200)">
            <return-response>
              <set-status code="400" reason="Content Safety Violation" />
            </return-response>
          </when>
        </choose>
      </when>
    </choose>
    
    <!-- Cost tracking headers -->
    <set-header name="x-contract-id" exists-action="override">
      <value>{{contract.id}}</value>
    </set-header>
    <set-header name="x-cost-center" exists-action="override">
      <value>{{contract.costCenter}}</value>
    </set-header>
  </inbound>
</policies>
```

### Runtime Validation

Gateway validates contracts on each request:

1. **Contract Exists**: Valid contract ID in request
2. **Not Expired**: Current date before `expiresAt`
3. **Status Active**: Contract status is `active`
4. **Rate Limits**: Current usage under quotas
5. **Content Safety**: Request passes safety checks
6. **Budget Check**: Current spend under budget

### Violation Handling

| Violation Type | Response | Action |
|----------------|----------|--------|
| **Rate Limit Exceeded** | `429 Too Many Requests` | Log, notify owner, queue for review |
| **Budget Exceeded** | `403 Forbidden` | Block, alert finance team |
| **Content Safety** | `400 Bad Request` | Log, flag for review |
| **Contract Expired** | `401 Unauthorized` | Prompt renewal |
| **Invalid Authentication** | `401 Unauthorized` | Log security event |

## Best Practices

### Contract Design

1. **Start Restrictive**: Begin with lower limits, increase based on usage
2. **Environment Segregation**: Separate contracts for dev/staging/prod
3. **Time-Bound**: Set expiration to force periodic review
4. **Cost Controls**: Always include budget limits with alerts
5. **Safety First**: Enable content safety by default

### Contract Governance

1. **Naming Conventions**: Use consistent patterns (`team-environment-purpose`)
2. **Version Control**: Store contract definitions in Git
3. **Review Cadence**: Quarterly access reviews for all contracts
4. **Automation**: Automate deployment through CI/CD
5. **Monitoring**: Dashboard for contract usage and violations

### Security Considerations

1. **Least Privilege**: Minimum necessary permissions
2. **Credential Rotation**: Automatic key rotation before expiry
3. **Audit Trail**: Log all contract changes
4. **Approval Workflow**: Multi-person approval for production
5. **Emergency Revocation**: Ability to instantly revoke access

## Architecture Best Practices

### Right-Sizing APIM Tier Selection

Select the APIM tier based on your governance and availability requirements — not just traffic volume.

| Decision Driver | Recommended Tier |
|----------------|-----------------|
| Evaluation / POC (< 2 weeks) | Developer |
| Single-region, non-critical | Standard |
| Multi-team, production, internal | Standard |
| Multi-region, zone redundancy required | Premium |
| External API consumers + enterprise SLA | Premium |
| Hybrid on-premises connectivity | Premium |

**Key insight:** Migrating from Standard to Premium requires a new deployment (not in-place upgrade). If you expect enterprise requirements within 6 months, start with Premium to avoid migration downtime.

**WAF Pillar: Reliability** — Premium tier provides 99.99% SLA with zone redundancy enabled.

### Spoke Design Patterns by Workload Type

Design spoke VNets to match workload isolation requirements:

```
Pattern A: Team-Isolated Spoke (Recommended for most cases)
  - One spoke per business unit / application team
  - Team owns all resources in their spoke
  - Hub provides shared APIM gateway and monitoring
  - Best for: Enterprise teams with separate budgets and compliance requirements

Pattern B: Environment-Isolated Spoke
  - One spoke per environment (dev, staging, prod)
  - All teams share a production spoke
  - Best for: Small organizations with few AI workloads

Pattern C: Compliance-Isolated Spoke
  - Separate spoke for regulated data (healthcare, finance)
  - Strict NSG rules and dedicated private endpoints
  - Best for: Mixed-sensitivity workloads requiring data segregation
```

**Recommendation:** Start with Pattern A. It provides the best balance of isolation and operational simplicity.

### Network Segmentation Strategy

Segment networks to limit blast radius of security incidents:

```
Hub VNet (10.0.0.0/16)
├── snet-apim (10.0.1.0/24)           — API Management only
├── snet-private-endpoints (10.0.2.0/24) — All PaaS private endpoints
├── snet-management (10.0.3.0/24)     — Bastion, jump servers
└── snet-monitoring (10.0.4.0/24)     — Optional: monitoring agents

Spoke VNet (10.x.0.0/24 per spoke)
├── snet-ai-workload (10.x.0.0/25)   — Compute for AI workload
└── snet-data (10.x.0.128/26)        — Databases, storage
```

**Rules:**
- Never put AI workload compute in the hub VNet
- Use /24 or larger for subnets that will host private endpoints (each endpoint = 1 IP)
- Reserve consecutive CIDR ranges across spokes (prevents overlap errors during scale-out)

## Security Best Practices

### Zero Trust Implementation Patterns

The AI Landing Zone implements Zero Trust through layered controls:

**Layer 1 — Identity (Never Trust Implicitly)**
- All API consumers authenticate with Entra ID tokens (no anonymous access in production)
- Managed identities for service-to-service authentication (no stored credentials)
- Conditional Access policies for human access to management plane

**Layer 2 — Network (Verify Explicitly)**
- All PaaS services behind private endpoints (no public network access)
- NSG rules on every subnet (default-deny posture)
- APIM in internal VNet mode (not accessible from internet)
- Azure Front Door or Application Gateway as internet-facing ingress

**Layer 3 — Data (Least Privilege Access)**
- RBAC on all resources (Key Vault, OpenAI, Foundry)
- PII anonymization before data reaches AI models
- Content Safety filtering on all model inputs/outputs
- Audit logging retained 90+ days

```bicep
// Enforce no public network access on all AI resources
resource openAiAccount 'Microsoft.CognitiveServices/accounts@2023-05-01' = {
  properties: {
    publicNetworkAccess: 'Disabled'        // * Always disable in production
    networkAcls: {
      defaultAction: 'Deny'
      virtualNetworkRules: []              // Access only via private endpoint
    }
  }
}
```

**WAF Pillar: Security** — Zero trust aligns with WAF Security principle of "assume breach."

### Secret Rotation Schedules

| Secret Type | Rotation Frequency | Automation Method |
|-------------|-------------------|-------------------|
| APIM subscription keys | 90 days | Key Vault rotation + APIM named value update |
| Managed identity (system-assigned) | Never | Azure-managed, no rotation needed |
| Service principal client secrets | 90 days | Key Vault rotation policy |
| TLS/SSL certificates | Before expiry (monitor 60 days prior) | App Service Certificates or Let's Encrypt |
| API backend API keys | Per vendor recommendation | Key Vault rotation |

**Implementation:**
```bash
# Configure Key Vault rotation policy (example: 90-day auto-rotation)
az keyvault key rotation-policy update \
  --vault-name <kv-name> \
  --name <key-name> \
  --value '{
    "lifetimeActions": [
      {
        "trigger": {"daysBeforeExpiry": 30},
        "action": {"type": "Rotate"}
      },
      {
        "trigger": {"daysBeforeExpiry": 7},
        "action": {"type": "Notify"}
      }
    ],
    "attributes": {"expiryTime": "P90D"}
  }'
```

### RBAC Least Privilege Patterns

Assign roles at the most specific scope with the minimum required permissions:

```
Principle: Grant the minimum role at the most specific scope

GOOD: Cognitive Services User on specific OpenAI account
BAD:  Contributor on entire resource group

Role Assignment Matrix for AI Landing Zone:

| Identity | Resource | Role |
|----------|----------|------|
| APIM System Identity | Key Vault | Key Vault Secrets User |
| APIM System Identity | OpenAI Account | Cognitive Services User |
| AI Workload Identity | OpenAI Account | Cognitive Services User |
| DevOps Service Principal | Resource Group | Contributor + User Access Admin |
| Platform Team | APIM Service | API Management Service Contributor |
| Workload Team | APIM Products | API Management Service Reader |
```

**Avoid these anti-patterns:**
- Do not assign `Owner` to service principals (use `Contributor` + `User Access Administrator`)
- Do not assign roles at subscription scope unless explicitly required
- Do not use legacy Key Vault access policies (use RBAC model)

## Performance Best Practices

### Caching Strategy for AI Workloads

Not all AI requests are unique. Implement tiered caching:

**Tier 1 — APIM Response Cache (deterministic requests)**
```xml
<!-- Cache identical requests for temperature=0 model calls -->
<cache-lookup vary-by-developer="false" vary-by-developer-groups="false">
  <vary-by-header>Accept</vary-by-header>
  <!-- Cache key includes the request body hash -->
  <vary-by-query-parameter>api-version</vary-by-query-parameter>
</cache-lookup>
<!-- Cache 5 minutes for static content queries -->
<cache-store duration="300" />
```

**Tier 2 — Redis Cache (semantic similarity)**
- Use Azure Cache for Redis for semantic caching (similar questions → cached answer)
- Requires embedding comparison logic (custom APIM policy or external service)
- Reduces OpenAI costs by 20-40% for FAQ-style workloads

**Tier 3 — Context Caching (long documents)**
- Use OpenAI Prompt Caching for repeated system prompts
- Automatically active for prompts > 1024 tokens that repeat across requests
- Reduces latency and costs on document analysis workloads

**WAF Pillar: Performance Efficiency** — Caching can reduce P95 latency from 3s to under 200ms for cached requests.

### Auto-Scaling Thresholds

```bicep
// Auto-scale APIM capacity (Premium tier only)
resource apimAutoscale 'Microsoft.Insights/autoscalesettings@2022-10-01' = {
  name: 'autoscale-apim'
  location: location
  properties: {
    enabled: true
    targetResourceUri: apimService.id
    profiles: [
      {
        name: 'Default Profile'
        capacity: { minimum: '1', maximum: '4', default: '1' }
        rules: [
          {
            // Scale out when capacity > 70%
            metricTrigger: {
              metricName: 'Capacity'
              metricResourceUri: apimService.id
              timeGrain: 'PT1M'
              statistic: 'Average'
              timeWindow: 'PT10M'
              timeAggregation: 'Average'
              operator: 'GreaterThan'
              threshold: 70
            }
            scaleAction: {
              direction: 'Increase'
              type: 'ChangeCount'
              value: '1'
              cooldown: 'PT20M'         // Wait 20 min before scaling again
            }
          }
          {
            // Scale in when capacity < 30%
            metricTrigger: {
              metricName: 'Capacity'
              operator: 'LessThan'
              threshold: 30
              timeWindow: 'PT60M'       // Scale in slowly to avoid thrashing
            }
            scaleAction: {
              direction: 'Decrease'
              type: 'ChangeCount'
              value: '1'
              cooldown: 'PT60M'
            }
          }
        ]
      }
    ]
  }
}
```

### Latency Optimization Techniques

1. **Deploy OpenAI in the same region as APIM** — cross-region adds 20-100ms per call
2. **Use private endpoints** — eliminates public internet routing (saves 10-50ms)
3. **Enable APIM compression** for large responses:
   ```xml
   <outbound>
     <set-header name="Accept-Encoding" exists-action="override">
       <value>gzip</value>
     </set-header>
   </outbound>
   ```
4. **Pre-warm model deployments** — send a warm-up request after deployment
5. **Use streaming** for long responses — reduces perceived latency significantly:
   ```python
   # Client-side streaming example
   response = client.chat.completions.create(
     model="gpt-4o",
     messages=[{"role": "user", "content": prompt}],
     stream=True  # Stream tokens as they're generated
   )
   ```

### Reserved Capacity Planning

Use Azure OpenAI Provisioned Throughput Units (PTUs) when:
- Monthly token volume is predictable (variance < 30%)
- Volume exceeds 10M tokens/month per model
- P99 latency requirements are strict (PTU provides consistent throughput)

PTU break-even calculation:
```
Pay-as-you-go cost: $0.005 / 1K input tokens × monthly_tokens
PTU cost: $2/hour × 730 hours × ptu_units (1 PTU ≈ 2,500 TPM for GPT-4o)

Break-even: typically at ~5-10M tokens/month for GPT-4o
```

### Monitoring and Alerting on Spend

```kql
// Daily token cost estimate by workload
ApiManagementGatewayLogs
| where TimeGenerated > ago(7d)
| extend
    inputTokens = toint(ResponseHeaders_x_prompt_tokens_s),
    outputTokens = toint(ResponseHeaders_x_completion_tokens_s)
| summarize
    TotalInputTokens = sum(inputTokens),
    TotalOutputTokens = sum(outputTokens),
    // GPT-4o pricing: $5/1M input, $15/1M output
    EstimatedCostUSD = (sum(inputTokens) / 1000000.0 * 5.0) + (sum(outputTokens) / 1000000.0 * 15.0)
  by ProductName, bin(TimeGenerated, 1d)
| order by EstimatedCostUSD desc
```

## Governance Best Practices

### AI Access Contract Versioning

Treat AI Access Contracts like software — version them:

```
Contract versioning convention: <name>-v<major>.<minor>
- Major version: Breaking changes (new required fields, model changes, rate limit reductions)
- Minor version: Additive changes (new optional fields, limit increases)

Examples:
  hr-chat-agent-v1.0  → Initial contract
  hr-chat-agent-v1.1  → Added PII protection (non-breaking)
  hr-chat-agent-v2.0  → Changed from GPT-4 to GPT-4o (breaking: model change)
```

**Deprecation pattern:**
1. Deploy new contract version alongside old version
2. Notify workload team with 30-day migration window
3. Monitor traffic shifting from v1 to v2 in APIM logs
4. Deactivate old contract version after migration complete

### Change Management Procedures

All production changes must follow:

1. **Change Request** — Describe impact, rollback plan, test evidence
2. **Staging Validation** — Deploy to staging, run validation checklist
3. **Change Window** — Schedule during low-traffic period (typically 2-6 AM UTC)
4. **Deployment** — Use Bicep `what-if` before applying:
   ```bash
   az deployment group what-if \
     --resource-group <prod-rg> \
     --template-file main.bicep \
     --parameters main.parameters.prod.bicepparam
   ```
5. **Verification** — Run post-deployment validation checklist
6. **Rollback** — If validation fails, redeploy previous version within 30 minutes

### Incident Response Playbooks

**Severity 1 (Complete Outage): AI API Unavailable**

```
T+0: Alert fires → On-call notified
T+5: Acknowledge alert, join incident bridge
T+10: Quick diagnosis:
  - Is APIM service healthy? (az apim show)
  - Are backends reachable? (test connectivity)
  - Are DNS names resolving? (nslookup from VNet)
T+20: Implement workaround if available (failover to secondary region)
T+30: Root cause hypothesis → implement fix
T+60: Verify resolution → close incident
T+120: Write incident report, update runbook
```

**Severity 2 (Degraded Performance): High Latency or Error Rate**
```
T+0: Alert fires → team notified via Teams
T+15: Identify affected component via Log Analytics queries
T+30: Apply fix (clear cache, adjust rate limits, restart unhealthy backend)
T+60: Verify metrics returning to normal
T+240: Root cause analysis → update monitoring thresholds
```

## Migration and Upgrade

### Zero-Downtime Deployment Patterns

Use blue-green deployment for all breaking APIM changes:

```
Blue environment: Current production (APIM v1 + OpenAI account A)
Green environment: New version (APIM v2 + OpenAI account B)

1. Deploy green environment alongside blue (same resource group, different names)
2. Validate green with test traffic (10%)
3. Shift traffic via APIM backend pool weights:
   90% blue, 10% green → 50/50 → 10% blue, 90% green → 100% green
4. Monitor for 24 hours at 100% green
5. Decommission blue
```

### Canary Release Approach for New AI Models

```xml
<!-- Canary policy: route 10% of traffic to new model -->
<choose>
  <when condition="@(new Random().NextDouble() < 0.10)">
    <!-- Canary: new model -->
    <set-backend-service backend-id="openai-gpt4o-new-deployment" />
    <set-header name="x-canary" exists-action="override">
      <value>true</value>
    </set-header>
  </when>
  <otherwise>
    <!-- Stable: existing model -->
    <set-backend-service backend-id="openai-gpt4o-stable" />
  </otherwise>
</choose>
```

Monitor canary vs. stable in Log Analytics:
```kql
ApiManagementGatewayLogs
| where TimeGenerated > ago(1h)
| extend isCanary = tostring(RequestHeaders_x_canary_s) == "true"
| summarize
    AvgLatency = avg(totalTime),
    ErrorRate = 100.0 * countif(responseCode >= 400) / count()
  by isCanary
```

## Decision Trees

### Should I use APIM Internal or External VNet Mode?

```
Q: Do external (internet) consumers need direct access to APIM?
├── YES → Use External VNet mode or no VNet (add WAF/Front Door for protection)
└── NO → Continue...

Q: Do workloads span multiple Azure subscriptions/VNets?
├── YES → Use Internal VNet mode (peering handles cross-VNet access)
└── NO → Continue...

Q: Is all traffic internal to Azure VNets?
├── YES → Use Internal VNet mode (most secure, private endpoints on all backends)
└── NO (hybrid on-prem) → Use Internal VNet mode + ExpressRoute
```

### Should I Use PTU or Pay-as-You-Go for OpenAI?

```
Q: Is your monthly token volume > 5M tokens for GPT-4o?
├── NO → Pay-as-you-go (lower cost at low volumes)
└── YES → Continue...

Q: Is volume predictable (< 30% variance month-to-month)?
├── NO → Pay-as-you-go (avoid paying for unused PTUs)
└── YES → Continue...

Q: Do you need guaranteed throughput (no throttling)?
├── YES → PTU (provides committed capacity, no throttling up to limit)
└── NO → Pay-as-you-go with rate limiting policies in APIM
```

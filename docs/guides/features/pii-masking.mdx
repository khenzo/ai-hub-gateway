# PII Masking

Using APIM to handle PII anonymization and deanonymization requests transparently for API requests that may contain PII data.

This can be important for scenarios where you want to ensure that PII data is not exposed to the backend services, while still allowing the client to receive a deanonymized response.

One example of needing such functionality is when you want to use Azure OpenAI API with PII data but outside the supported/allowed regions (geo-restrictions), but you want to ensure that the PII data is not sent to the backend service. In this case, you can use APIM to handle the PII anonymization and deanonymization transparently.

## Key Updates to PII Handling Framework

The PII handling framework has been enhanced with the following key features:

1. **Managed Identity Authentication**: Authentication to Azure AI Language Services now uses managed identity instead of API keys, improving security.

2. **Regex Pattern Processing**: Extend Azure PII detection Natural Language Processing with regex pattern processing for custom PII detection prior to calling the Language Service.

3. **Additional Configuration Options**: New configuration options for language detection and more customizable PII identification.

4. **Event Hub Logging**: Comprehensive logging to Event Hub for auditing, compliance, and testing.

5. **PII Blocking**: New capability to completely block requests containing PII data with a 400 Bad Request response, useful for strict compliance scenarios where no PII should reach backend services.

## Process Flow

```
Client Request
     │
     ▼
┌──────────────────────────────────────┐
│  APIM Gateway (Inbound Processing)   │
│  • Send to Azure Language Service    │
│  • Return PII entities               │
│  • Apply regex patterns              │
│  • Create PII mappings               │
│  • Replace PII with placeholders     │
└──────────────────┬───────────────────┘
                   │ Anonymized Request
                   ▼
         ┌──────────────────┐
         │  Backend API     │
         └──────────────────┘
                   │ Response
                   ▼
┌──────────────────────────────────────┐
│  APIM Gateway (Outbound Processing)  │
│  • Store response body               │
│  • Apply deanonymization             │
│  • Log to Event Hub (if enabled)     │
└──────────────────┬───────────────────┘
                   │ Deanonymized Response
                   ▼
               Client
```

## Implementation Components

### PII Detection Methods

- Azure Language Service PII detection (using Natural Language Processing - NLP)
- Custom regex pattern matching (provided by the user)
- Configurable confidence thresholds
- Category exclusions (e.g., excluding PersonType as usually LLM request context containers `user` and `assistant` are not PII data but rather part of the request context)

### Regex Pattern Support

Example of extending PII detection with Regex patterns for common PII types:

```json
{
    "patterns": [
        {
            "pattern": "\\b\\d{4}[- ]?\\d{4}[- ]?\\d{4}[- ]?\\d{4}\\b",
            "category": "CreditCard"
        },
        {
            "pattern": "\\b[A-Z]{2}\\d{6}[A-Z]\\b",
            "category": "PassportNumber"
        },
        {
            "pattern": "\\b\\d{3}[-]?\\d{4}[-]?\\d{7}[-]?\\d{1}\\b",
            "category": "EMIRATESID"
        },
        {
            "pattern": "\\b\\d{5,}\\b",
            "category": "CUSTOMERID"
        }
    ]
}
```

## Configuration

### Enable PII Masking

PII masking is controlled via the `enableAIGatewayPiiRedaction` parameter:

```bicep
param enableAIGatewayPiiRedaction = true
```

### PII Service Configuration

```bicep
param languageServiceSkuName = 'S'  // Standard tier
param languageServiceExternalNetworkAccess = 'Disabled'  // Secure deployment
```

### Custom Regex Patterns

Upload your regex patterns to the configuration store or include them in your deployment parameters.

## Understanding PII Anonymization API

The following is an example that sends a request to Azure Language Service PII Anonymization API:

```bash
curl --request POST \
  --url 'https://{language-service-url}/language/:analyze-text?api-version=2022-05-01&showStats=showStats' \
  --header 'Content-Type: application/json' \
  --header 'Ocp-Apim-Subscription-Key: {key}' \
  --data '{
  "kind": "PiiEntityRecognition",
  "analysisInput": {
    "documents": [
      {
        "id": "1",
        "language": "en",
        "text": "Call our office at 312-555-1234, or send an email to support@contoso.com."
      }
    ]
  },
  "parameters": {
    "domain": "phi",
    "modelVersion": "latest"
  }
}'
```

## Event Hub Logging

PII processing events are logged to Event Hub for auditing:

| Log Type | Description |
|----------|-------------|
| `pii_detected` | PII entities found in request |
| `pii_masked` | PII entities replaced with placeholders |
| `pii_blocked` | Request blocked due to PII detection |
| `pii_deanonymized` | Response deanonymized |

## PII Blocking Mode

Enable strict PII blocking for compliance scenarios:

```bicep
param piiBlockingMode = 'strict'  // 'permissive' | 'strict'
```

In **strict mode**, requests containing PII return:
```json
{
  "error": {
    "code": "PII_DETECTED",
    "message": "Request contains PII data and has been blocked per compliance policy.",
    "details": {
      "categories": ["Person", "Email"]
    }
  }
}
```

## Best Practices

1. **Use managed identity** for Language Service authentication
2. **Configure appropriate confidence thresholds** - balance detection accuracy vs false positives
3. **Add custom regex patterns** for organization-specific PII types
4. **Enable Event Hub logging** for compliance auditing
5. **Test thoroughly** before enabling in production
6. **Document excluded categories** for your use case
7. **Monitor PII detection rates** and adjust patterns as needed

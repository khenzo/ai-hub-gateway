# Networking Architecture

The AI Landing Zone implements a secure, scalable network architecture using Azure's hub-and-spoke topology with AI-specific connectivity patterns.

## Hub and Spoke Overview

The Citadel architecture separates the central governance hub from workload-specific landing zones:

## Network Topology

## Connectivity Patterns

### Spoke-to-Hub Communication

Landing zones communicate with the Citadel hub for:

- **API Gateway**: All AI requests route through the hub's APIM
- **Logging**: Diagnostic data flows to central Log Analytics
- **Policy**: Configuration and compliance status reporting

### Spoke-to-Spoke Communication

Direct spoke-to-spoke communication is typically **not required** for AI workloads. Each spoke operates independently, accessing shared services through the hub.

<Note>
  If cross-spoke communication is needed, route traffic through the hub firewall for inspection and logging.
</Note>

## Security Features

### Network Isolation

- **Private Endpoints**: All AI services use private endpoints (no public internet exposure)
- **Network Segmentation**: Subnets isolate different workload types
- **Micro-segmentation**: NSGs provide subnet-level traffic control

### Traffic Inspection

<AccordionGroup>
  <Accordion title="Azure Firewall">
    - Centralized egress filtering
    - FQDN filtering for outbound traffic
    - Threat intelligence integration
    - Application rules for HTTP/S traffic
  </Accordion>
  
  <Accordion title="Network Security Groups">
    - Subnet-level access control
    - Service tag support for Azure services
    - Application security groups for workload grouping
  </Accordion>
</AccordionGroup>

### DNS Configuration

Private DNS zones enable name resolution for private endpoints:

```
privatelink.openai.azure.com
privatelink.cognitiveservices.azure.com
privatelink.search.windows.net
privatelink.api.azureml.ms
```

These zones are linked to all VNets to enable cross-network resolution.

## Deployment Patterns

### Pattern 1: Existing Hub

Deploy Citadel components into your existing Azure hub network:

**Benefits:**
- Leverages existing connectivity (VPN/ExpressRoute)
- Consistent with current network architecture
- Simpler DNS and routing configuration

**Considerations:**
- Must meet Citadel's network requirements
- May require NSG rule updates
- IP address space planning

### Pattern 2: New Hub

Deploy a dedicated Citadel hub network:

**Benefits:**
- Clean separation of AI infrastructure
- Custom network sizing and segmentation
- Independent lifecycle management

**Considerations:**
- Additional VNet peering configuration
- Separate connectivity requirements
- Higher operational overhead

### Pattern 3: Virtual WAN

Integrate Citadel with Azure Virtual WAN:

**Benefits:**
- Simplified large-scale connectivity
- Centralized routing and security
- Native SD-WAN integration

**Considerations:**
- Requires Virtual WAN expertise
- Custom routing policies needed
- May impact existing WAN architecture

# Azure Landing Zone Integration

This guide provides comprehensive instructions for integrating Citadel AI Landing Zone with your existing Azure Landing Zone (ALZ) environment. Following this guidance ensures seamless integration with shared services while maintaining enterprise governance and security standards.

## Overview

Citadel is designed as a **supplemental AI landing zone** that extends your existing Azure Landing Zone. This approach:

- **Reuses shared infrastructure** - Leverages existing identity, network, and monitoring
- **Follows ALZ patterns** - Consistent with Cloud Adoption Framework guidance
- **Accelerates deployment** - 50-60% faster than standalone deployment
- **Maintains governance** - Inherits enterprise policies and security controls

## Integration Points

### 1. Networking Integration

#### Hub VNet Peering

Citadel integrates with your existing hub VNet via VNet peering, enabling secure connectivity to shared network infrastructure.

**Prerequisites:**
- Existing hub VNet deployed
- Non-overlapping IP address space identified for CGH
- Network security requirements documented

**Configuration:**

```bicep
// Hub to CGH peering
resource hubToCghPeering 'Microsoft.Network/virtualNetworks/virtualNetworkPeerings@2023-09-01' = {
  name: '${hubVnetName}/peering-to-${cghVnetName}'
  properties: {
    allowVirtualNetworkAccess: true
    allowForwardedTraffic: true
    allowGatewayTransit: true    // Hub shares gateway with CGH
    useRemoteGateways: false
    remoteVirtualNetwork: {
      id: cghVnet.id
    }
  }
}

// CGH to Hub peering
resource cghToHubPeering 'Microsoft.Network/virtualNetworks/virtualNetworkPeerings@2023-09-01' = {
  name: '${cghVnetName}/peering-to-${hubVnetName}'
  properties: {
    allowVirtualNetworkAccess: true
    allowForwardedTraffic: false
    allowGatewayTransit: false
    useRemoteGateways: true      // CGH uses hub gateway for hybrid
    remoteVirtualNetwork: {
      id: hubVnet.id
    }
  }
}
```

**Traffic Flow:**

| Source | Destination | Path | Purpose |
|--------|-------------|------|---------|
| CAS Spoke | CGH Services | Via Hub | AI gateway access |
| CGH | On-premises | Via Hub Gateway | Hybrid connectivity |
| CGH | Internet | Via Hub Firewall | Secure egress |
| CAS Spoke | CAS Spoke | Via Hub Firewall | Cross-team communication |

### 2. Identity Integration

#### Entra ID Integration

Citadel leverages your existing Entra ID tenant for authentication and authorization.

**Integration Points:**

| Component | Entra ID Feature | Usage |
|-----------|------------------|-------|
| API Management | OAuth 2.0 / OIDC | JWT validation for AI requests |
| Managed Identities | System/User-assigned | Service-to-service auth |
| Access Packages | Identity Governance | Agent access lifecycle |
| Conditional Access | Security policies | Multi-factor auth, device compliance |

**APIM OAuth Configuration:**

```xml
<!-- APIM Policy: JWT Validation -->
<validate-jwt
    header-name="Authorization"
    failed-validation-httpcode="401"
    require-expiration-time="true"
    require-scheme="Bearer">
    <openid-config url="https://login.microsoftonline.com/{tenant-id}/v2.0/.well-known/openid-configuration" />
    <audiences>
        <audience>api://cgh-ai-gateway</audience>
    </audiences>
    <required-claims>
        <claim name="roles" match="any">
            <value>AIUser</value>
            <value>AIAdmin</value>
        </claim>
    </required-claims>
</validate-jwt>
```

### 4. Governance Integration

#### Azure Policy

Citadel inherits policies from the platform landing zone and adds AI-specific policies.

**Policy Hierarchy:**

```
Management Group: Contoso-ALZ
├── Initiative: ALZ-Platform-Baseline
│   ├── Require NSG on subnet
│   ├── Require private endpoint
│   ├── Require diagnostic settings
│   └── [Other platform policies]
│
└── Management Group: Contoso-AI
    └── Initiative: Citadel-AI-Governance
        ├── Require content safety on APIM
        ├── Require PII detection
        ├── Enforce token quotas
        └── Require AI audit logging
```

**Custom Policy Definitions:**

```json
{
  "policyRule": {
    "if": {
      "allOf": [
        {
          "field": "type",
          "equals": "Microsoft.ApiManagement/service"
        },
        {
          "field": "Microsoft.ApiManagement/service/policies",
          "exists": "false"
        }
      ]
    },
    "then": {
      "effect": "deny"
    }
  }
}
```

## Deployment Patterns

### Pattern A: Greenfield Deployment

**When to Use:**
- No existing Azure Landing Zone
- Starting cloud journey
- Want integrated AI governance from start

**Deployment Steps:**

1. **Deploy ALZ Foundation** (Week 1)
   ```bash
   # Deploy ALZ using Azure Landing Zone accelerator
   az deployment mg create \
     --name alz-deployment \
     --management-group-id "contoso-landing-zones" \
     --template-file alz-template.json \
     --parameters environment=Production
   ```

2. **Validate Platform Services** (Week 1-2)
   - Verify Hub VNet connectivity
   - Test Log Analytics workspace
   - Confirm Entra ID integration

3. **Deploy Citadel CGH** (Week 2-3)
   ```bash
   # Deploy Citadel integrated with new ALZ
   az deployment sub create \
     --name citadel-deployment \
     --location eastus \
     --template-file citadel-main.bicep \
     --parameters hubVnetResourceId=/subscriptions/.../hub-vnet
   ```

4. **Configure Integration** (Week 3-4)
   - VNet peering
   - Private DNS linking
   - Policy assignment
   - Monitoring configuration

5. **Deploy First CAS Spoke** (Week 4-5)
   - Validate end-to-end flow
   - Test governance policies
   - Document patterns

### Pattern B: Brownfield Deployment

**When to Use:**
- Existing Azure Landing Zone in production
- Active workloads running
- Need to add AI governance

**Deployment Steps:**

1. **Assess Existing ALZ** (Week 1)
   - Document current hub VNet configuration
   - Identify IP address space availability
   - Review existing policies and RBAC
   - Assess monitoring infrastructure

2. **Plan Integration** (Week 1-2)
   ```markdown
   ## Integration Planning Checklist
   
   ### Network
   - [ ] Identify available IP range for CGH
   - [ ] Document firewall rules needed
   - [ ] Plan private DNS integration
   - [ ] Assess ExpressRoute/VPN capacity
   
   ### Identity
   - [ ] Review Entra ID structure
   - [ ] Plan RBAC assignments
   - [ ] Identify Conditional Access impact
   
   ### Governance
   - [ ] Review existing policies
   - [ ] Plan AI-specific policy additions
   - [ ] Document tagging strategy alignment
   
   ### Monitoring
   - [ ] Confirm Log Analytics workspace access
   - [ ] Plan dashboard integration
   - [ ] Review alerting infrastructure
   ```

3. **Deploy CGH in Parallel** (Week 2-3)
   - Deploy to separate subscription
   - Configure VNet peering to hub
   - Do not route production traffic yet

4. **Validate Integration** (Week 3-4)
   - Test connectivity
   - Verify logging
   - Validate policies
   - Performance testing

5. **Pilot Migration** (Week 4-6)
   - Migrate non-critical workload
   - Monitor for issues
   - Refine configuration

6. **Full Rollout** (Week 6-8)
   - Migrate remaining workloads
   - Decommission direct model access
   - Update documentation

### Pattern C: Hybrid Deployment

**When to Use:**
- Partial ALZ implementation
- Different maturity levels across subscriptions
- Need to bridge gaps

**Approach:**

```
┌─────────────────────────────────────────────────────────────┐
│              Hybrid Deployment Model                         │
│                                                              │
│   Subscription A (Mature ALZ)                                │
│   ├── Full platform services                                │
│   └── CGH integrates directly                               │
│                                                              │
│   Subscription B (Partial ALZ)                              │

│   Subscription C (No ALZ)                                   │
│   ├── CGH deploys mini-ALZ                                  │
```

## Design Considerations

### IP Address Space Planning

**Recommendation:** Plan for 3-5 years of growth

| Network | Size | Hosts | Purpose |
|---------|------|-------|---------|
| Hub VNet | /22 | ~1,000 | Shared services |
| CGH VNet | /23 | ~500 | Governance hub |
| CAS VNet (each) | /22 | ~1,000 | AI workloads per BU |
| Reserved | /16 | ~65,000 | Future expansion |

**Address Space Example:**

```
10.0.0.0/16 - Corporate Network
├── 10.0.0.0/22 - Hub VNet (ALZ)
├── 10.0.4.0/23 - CGH VNet
├── 10.0.6.0/22 - CAS-BU1 VNet
├── 10.0.10.0/22 - CAS-BU2 VNet
├── 10.0.14.0/22 - CAS-BU3 VNet
└── 10.0.18.0/18 - Reserved
```

### Policy Inheritance and Exemptions

**Inheritance Model:**

```
Management Group: Contoso-Root
└── Management Group: Contoso-Platform
    └── Policy: Require NSG on Subnet [Enforce]
    └── Subscription: Platform-Prod
        └── CGH inherits policy
    └── Subscription: Platform-Dev
        └── CGH inherits policy
        └── Exemption: Dev environment
            └── Policy: Require NSG [Audit only]
```

**When to Use Exemptions:**

| Scenario | Exemption Type | Duration |
|----------|---------------|----------|
| Development environment | Audit instead of Deny | Permanent |
| Emergency change | Waiver | 7 days |
| Legacy system migration | Audit | 90 days |
| Third-party requirements | Waiver | Permanent |

### RBAC Delegation Models

**Model A: Centralized**

- Platform team manages all Citadel resources
- Application teams submit requests
- Best for: Small teams, high governance requirements

**Model B: Delegated**

- Platform team manages CGH
- Application teams manage CAS spokes
- Best for: Large enterprises, autonomous teams

**Model C: Federated**

- Platform team manages shared services
- Business unit admins manage CGH instances
- Application teams manage spokes
- Best for: Multi-tenant, global organizations

### Cost Attribution Strategies

**Approach 1: Subscription-Based**

- Each CAS spoke in separate subscription
- Natural cost boundary
- Simple but many subscriptions

**Approach 2: Tag-Based**

- Use `BusinessUnit` and `CostCenter` tags
- Query costs via Cost Management APIs
- More flexible, requires discipline

**Approach 3: Hybrid**

- Production: Subscription-based
- Non-production: Tag-based
- Balances simplicity and flexibility

## Troubleshooting

### Common Issues

**Issue 1: VNet Peering Not Connecting**

```bash
# Check peering state
az network vnet peering show \
  --name cgHub-to-hub \
  --resource-group cgHub-rg \
  --vnet-name cgHub-vnet \
  --query peeringState

# Expected: "Connected"
# If "Initiated": Check hub-side peering
```

**Issue 2: Private DNS Not Resolving**

```bash
# Verify VNet link exists
az network private-dns link vnet list \
  --resource-group hub-rg \
  --zone-name privatelink.azure-api.net

# Verify record exists
az network private-dns record-set a show \
  --resource-group hub-rg \
  --zone-name privatelink.azure-api.net \
  --name cgHub-apim
```

**Issue 3: Logs Not Appearing in Log Analytics**

```bash
# Check diagnostic settings
az monitor diagnostic-settings list \
  --resource $(az apim show --name cgHub-apim --query id -o tsv)

# Verify workspace access
az monitor log-analytics workspace get-shared-keys \
  --resource-group hub-rg \
  --name hub-workspace
```

**Issue 4: Policy Conflicts**

```bash
# Check policy compliance
az policy state summarize \
  --resource-group cgHub-rg \
  --query "results.nonCompliantResources"

# View assignment details
az policy assignment list \
  --resource-group cgHub-rg \
  --query "[].{Name:name, Policy:policyDefinitionId}"
```

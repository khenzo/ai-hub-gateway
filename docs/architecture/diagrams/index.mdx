## 1. Citadel 4-Layer Architecture

### Overview

The Foundry Citadel Platform follows a **4-layer architecture** that provides end-to-end AI governance through separation of concerns with unified oversight.

![4-Layer Architecture](/assets/citadel-governance-hub-v1.png)

**Layer Relationships:**

| Layer | Name | Responsibility | Key Components |
|-------|------|----------------|----------------|
| 🔷 **Layer 1** | **Governance Hub** | Runtime enforcement | Azure API Management, API Center, Content Safety, Language Service |
| 🔶 **Layer 2** | **AI Control Plane** | Observability & compliance | Microsoft Foundry Control Plane, AI Evaluations, Fleet Operations |
| 🟢 **Layer 3** | **Agent Identity (Agent 365)** | Agent identity & lifecycle | Microsoft Entra ID, Shadow Agent Detection, Access Packages |
| 🛡️ **Layer 4** | **Security Fabric** | Unified protection | Microsoft Defender, Purview, Entra ID |

**Interactions:**

- **Layer 1 → Layer 2**: Governance Hub emits telemetry and audit logs to Control Plane
- **Layer 2 → Layer 1**: Control Plane provides policy definitions and evaluation results
- **Layer 1 ↔ Layer 3**: Gateway validates agent identities at runtime
- **Layer 4 → All Layers**: Security Fabric provides threat intelligence and protection across all layers

## 2. Hub-Spoke Topology

### 2.1 High-Level Topology

The hub-spoke architecture provides centralized governance while enabling distributed AI development.

```
┌─────────────────────────────────────────────────────────┐
│                    Azure Landing Zone                    │
│  ┌─────────────────────────────────────────────────┐    │
│  │         Citadel Governance Hub (CGH)             │    │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────┐  │    │
│  │  │   API Mgmt  │  │  API Center │  │Content  │  │    │
│  │  │   (Gateway) │  │  (Registry) │  │ Safety  │  │    │
│  │  └──────┬──────┘  └─────────────┘  └─────────┘  │    │
│  │         │                                        │    │
│  │  ┌─────────────┐  ┌─────────────┐               │    │
│  │  │Log Analytics│  │  Event Hub  │               │    │
│  │  │   Cosmos DB │  │   Logic App │               │    │
│  │  └─────────────┘  └─────────────┘               │    │
│  └─────────┬───────────────────────────────────────┘    │
│            │                                             │
│            │ VNet Peering                                 │
│            ▼                                             │
│  ┌─────────────────┐    ┌─────────────────┐              │
│  │   CAS Spoke 1   │    │   CAS Spoke 2   │              │
│  │ (Business Unit) │    │ (Business Unit) │              │
│  │ ┌─────────────┐ │    │ ┌─────────────┐ │              │
│  │ │AI Foundry   │ │    │ │AI Foundry   │ │              │
│  │ │Container App│ │    │ │Container App│ │              │
│  │ └─────────────┘ │    │ └─────────────┘ │              │
│  └─────────────────┘    └─────────────────┘              │
└─────────────────────────────────────────────────────────┘
```

**Key Benefits:**

- **Centralized governance**: All AI traffic flows through hub for policy enforcement
- **Network isolation**: Spokes are isolated but connected via peering
- **Cost attribution**: Clear boundaries for chargeback per business unit
- **Scalability**: Add spokes without impacting hub operations

### 2.2 Hub-Spoke Network Flow

**CGH Hub Network:**

| Subnet | Purpose | Services |
|--------|---------|----------|
| `GatewaySubnet` | VPN/ExpressRoute Gateway | VPN Gateway, ER Gateway |
| `AzureFirewallSubnet` | Central firewall | Azure Firewall |
| `CGHSubnet` | Core services | APIM, API Center, Content Safety |
| `MonitoringSubnet` | Observability | Log Analytics, Application Insights |

**CAS Spoke Networks:**

| Subnet | Purpose | Services |
|--------|---------|----------|
| `ComputeSubnet` | Agent workloads | AI Foundry, Container Apps, AKS |
| `DataSubnet` | Data services | Cosmos DB, Storage Account, AI Search |
| `IntegrationSubnet` | Service integration | Service Bus, Event Grid |

**Connectivity Patterns:**

- Hub to Spoke: VNet peering with gateway transit
- Spoke to Internet: Forced tunneling through hub firewall
- Spoke to Spoke: Via hub (no direct peering)
- On-premises: Via hub VPN/ExpressRoute gateway

## 3. Deployment Architecture

### 3.1 Part of Hub Network Pattern

In this pattern, the Citadel Governance Hub is deployed **within the existing hub VNet** of your Azure Landing Zone.

```
┌────────────────────────────────────────────────────────┐
│                  Hub Virtual Network                   │
│                                                        │
│   ┌─────────────────┐    ┌─────────────────────────┐  │
│   │  Existing Hub   │    │  Citadel Governance Hub │  │
│   │   Services      │    │        (CGH)            │  │
│   │ ┌───────────┐   │    │  ┌─────────────────┐    │  │
│   │ │Firewall   │   │    │  │  API Management │    │  │
│   │ │VPN GW     │   │    │  │  API Center     │    │  │
│   │ │DNS        │   │    │  │  Content Safety │    │  │
│   │ └───────────┘   │    │  └─────────────────┘    │  │
│   └────────┬────────┘    └───────────┬─────────────┘  │
│            │                         │                │
│            └──────────┬──────────────┘                │
│                       │                               │
│              ┌────────┴────────┐                      │
│              │   Shared NSGs   │                      │
│              │   Route Tables  │                      │
│              └─────────────────┘                      │
└────────────────────────────────────────────────────────┘
                              │
                              │ VNet Peering
                              ▼
                    ┌──────────────────────┐
                    │   Spoke Networks     │
                    │ (Agent Workloads)    │
                    └──────────────────────┘
```

**When to Use:**

- Existing hub has capacity
- Simplified network management
- Lower latency between hub services
- Smaller scale deployments (< 50 spokes)

### 3.2 Part of Spoke Network Pattern

In this pattern, the Citadel Governance Hub is deployed in a **dedicated spoke VNet** that connects to the hub VNet via peering.

```
┌─────────────────────────────────────────────────────────┐
│                  Hub Virtual Network                    │
│                                                         │
│   ┌──────────────┐    ┌──────────────┐                 │
│   │   Firewall   │    │   VPN GW     │                 │
│   │   DNS Zones  │    │   Route Table│                 │
│   └──────────────┘    └──────────────┘                 │
└─────────┬───────────────────────────────────────────────┘
          │
          │ VNet Peering
          ▼
┌─────────────────────────────────────────────────────────┐
│              Citadel Governance Hub VNet                │
│                                                         │
│   ┌───────────────────────────────────────────────┐    │
│   │          Citadel Governance Hub               │    │
│   │  ┌─────────────┐  ┌─────────────────────┐     │    │
│   │  │ API Mgmt    │  │   Monitoring Stack  │     │    │
│   │  │ API Center  │  │ ┌─────┐ ┌─────────┐ │     │    │
│   │  │ Content Sfy │  │ │Cosmos│ │Log A    │ │     │    │
│   │  └─────────────┘  │ │EventH│ │App Ins  │ │     │    │
│   │                   │ └─────┘ └─────────┘ │     │    │
│   │                   └─────────────────────┘     │    │
│   └───────────────────────────────────────────────┘    │
└─────────────────┬───────────────────────────────────────┘
                  │
                  │ VNet Peering (via Hub)
                  ▼
┌─────────────────────────────────────────────────────────┐
│                 Agent Spoke Networks                    │
│                                                         │
│   ┌──────────────┐    ┌──────────────┐                 │
│   │   CAS Spoke  │    │   CAS Spoke  │                 │
│   │     (BU1)    │    │     (BU2)    │                 │
│   └──────────────┘    └──────────────┘                 │
└─────────────────────────────────────────────────────────┘
```

**When to Use:**

- Large-scale deployments (100+ spokes)
- Strong separation of concerns required
- Hub capacity constraints
- Multi-region deployments

## 5. Data Flow Diagrams

### 5.1 AI Request/Response Flow

```
┌────────────┐     ┌─────────────┐     ┌──────────────┐     ┌────────────┐
│   Agent    │     │   APIM AI   │     │   Content    │     │  Backend   │
│  (Spoke)   │────▶│   Gateway   │────▶│   Safety     │────▶│   LLM      │
└────────────┘     └─────────────┘     └──────────────┘     └────────────┘
      │                   │                   │                   │
      │                   │                   │                   │
      │ 1. Request        │ 2. Authenticate   │ 3. Safety Check   │ 4. Process
      │    (Prompt)       │    Authorize      │    Filter         │    Generate
      │                   │                   │                   │
      │                   │                   │                   │
      │ 8. Response       │ 7. Transform      │ 6. Safety Check   │ 5. Return
      │    (Answer)       │    Return         │    Filter         │    Response
      │                   │                   │                   │

Step-by-Step:
─────────────
1. Agent sends AI request with prompt and context
2. APIM validates identity, checks rate limits, applies routing rules
3. Content Safety scans prompt for harmful content
4. Backend LLM processes the request and generates response
5. LLM returns generated response to gateway
6. Content Safety scans response for harmful content
7. APIM transforms response, adds headers, logs telemetry
8. Response returned to agent with governance metadata
```

**Key Flow Points:**

| Step | Component | Action |
|------|-----------|--------|
| 1 | Agent | Sends request via gateway endpoint |
| 2 | APIM | JWT validation, rate limiting, routing |
| 3 | Content Safety | Prompt shield, content filtering |
| 4 | Backend | LLM inference, tool execution |
| 5-8 | Reverse flow | Response processing and return |

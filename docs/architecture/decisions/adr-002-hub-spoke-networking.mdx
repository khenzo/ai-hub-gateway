# ADR-002: Hub-Spoke Networking

## Status

**Accepted**

## Context

Deploying AI at enterprise scale requires a networking architecture that balances:

1. **Centralized Governance** - Unified policy enforcement and observability
2. **Distributed Development** - Teams need autonomy to build and deploy AI solutions
3. **Network Security** - Isolation between workloads, controlled connectivity
4. **Scalability** - Support growth from 10 to 10,000+ AI agents
5. **Cost Management** - Clear attribution and shared infrastructure efficiency

The Citadel Governance Hub must serve as a central control plane while supporting multiple business units and use cases across the organization.

### Requirements

We need a network architecture that provides:

- **Centralized services** - AI gateway, monitoring, shared AI services
- **Workload isolation** - Business units separated but connected
- **Controlled connectivity** - Spokes can reach hub, limited spoke-to-spoke
- **Scalability** - Add new spokes without reconfiguring existing infrastructure
- **Security** - Defense in depth, traffic inspection, audit trails
- **Cost attribution** - Clear boundaries for chargeback

### Alternative Approaches Considered

We evaluated five network topologies:

- NSG complexity at scale (100+ rules)
- Limited scalability (subscription limits)

**Pros:**
- Maximum isolation
- Simple security model
- No shared infrastructure dependencies

**Cons:**
- No centralized governance
- Duplicate infrastructure in each VNet
- No shared AI services or registry
- Manual configuration everywhere
- Violates core Citadel value proposition

**Verdict:** вқҢ **Rejected** - Defeats purpose of unified governance

#### Alternative 4: Single VNet with NSG Isolation

**Description:** Use Network Security Groups (NSGs) to isolate workloads within a single large VNet.

**Pros:**
- Simpler than peering
- Existing Azure Landing Zone pattern
- Familiar to network teams

**Cons:**
- NSG rules become unmanageable at scale
- No traffic inspection between subnets
- Limited network-layer controls
- Difficult to audit and troubleshoot
- Doesn't provide true isolation

**Verdict:** вқҢ **Rejected** - Doesn't provide adequate isolation

## Decision

We will implement a **hub-spoke network topology** with the Citadel Governance Hub (CGH) as the hub and Citadel Agent Spokes (CAS) as spokes.

### Rationale

#### 1. Separation of Concerns with Unified Oversight

The hub-spoke model embodies Citadel's core architectural principle:

| Aspect | Hub (CGH) | Spokes (CAS) |
|--------|-----------|--------------|
| **Responsibility** | Governance, security, observability | AI development, business logic |
| **Teams** | Platform/DevOps | Application developers |
| **Scale** | Few deployments (1-3 per env) | Many deployments (10-1000+) |
| **Change Rate** | Stable, controlled | Rapid, iterative |
| **Services** | Shared infrastructure | Workload-specific resources |

This separation allows:
- Platform teams to focus on governance infrastructure
- Application teams to focus on AI innovation
- Clear ownership boundaries
- Independent scaling and updates

#### 2. Proven Azure Landing Zone Pattern

Hub-spoke is the recommended topology for Azure Landing Zones:

- **Microsoft CAF Alignment** - Follows Cloud Adoption Framework guidance
- **Enterprise Adoption** - Used by thousands of organizations
- **Well-Architected** - Validated for reliability, security, and performance
- **Tooling Support** - Native support in Azure Portal, CLI, Bicep, Terraform

This ensures Citadel integrates seamlessly with existing enterprise architectures.

#### 3. Network Isolation with Connectivity

VNet peering provides the right balance:

```
Hub-Spoke Connectivity:
=======================

Spoke вҶ’ Hub: вң… Allowed (reach shared services)
Hub вҶ’ Spoke: вң… Allowed (management, monitoring)
Spoke вҶ’ Spoke: вқҢ Blocked (via hub firewall)
Spoke вҶ’ Internet: вң… Via hub (forced tunneling)
On-Prem вҶ’ Spoke: вң… Via hub (ExpressRoute/VPN)
```

**Traffic Flow:**

1. **Agent вҶ’ Gateway** (Spoke вҶ’ Hub): AI requests route to CGH for governance
2. **Gateway вҶ’ Backend** (Hub вҶ’ Internet/Private): Forward to LLM endpoints
3. **Response Path** (Hub вҶ’ Spoke): Return through hub for logging
4. **Cross-Spoke** (Spoke вҶ’ Hub вҶ’ Spoke): Via hub firewall if needed

#### 4. Centralized Security Enforcement

All traffic passes through the hub, enabling:

- **Azure Firewall** - Central traffic inspection
- **NSG Flow Logs** - Network traffic audit trails
- **DDoS Protection** - Standard protection at hub
- **Private DNS** - Centralized name resolution
- **Private Endpoints** - Secure access to PaaS services

#### 5. Scalability from 10 to 10,000 Agents

The hub-spoke model scales efficiently:

| Scale | Hubs | Spokes | Peering Count |
|-------|------|--------|---------------|
| Small | 1 | 10 | 10 |
| Medium | 2 (prod/non-prod) | 50 | 100 |
| Large | 3 (prod/staging/dev) | 500 | 1,500 |
| Enterprise | 5+ (regional) | 10,000+ | 50,000+ |

**Scaling Considerations:**
- Hub can support 500+ spokes per region
- Multiple hubs for global deployments
- Gateway Transit allows spokes to share VPN/ER
- No reconfiguration needed when adding spokes

#### 6. Cost Attribution and Chargeback

Clear cost boundaries enable chargeback:

| Resource | Location | Attribution |
|----------|----------|-------------|
| CGH Infrastructure | Hub | Platform/shared cost |
| AI Gateway Usage | Hub | By spoke (via subscription keys) |
| Agent Workloads | Spoke | Business unit cost |
| Data Storage | Spoke | Business unit cost |
| Network Transfer | Hub | Platform/shared cost |

#### 7. Deployment Patterns

We support two hub-spoke deployment patterns:

**Pattern A: CGH in Hub VNet**
```
Existing Hub VNet (ALZ)
в”ңв”Җв”Җ Gateway Subnet
в”ңв”Җв”Җ Firewall Subnet
в”ңв”Җв”Җ CGH Subnet (NEW) вҶҗ CGH deployed here
в””в”Җв”Җ Shared Services
```
- Simpler, lower latency
- For: < 50 spokes, existing hub with capacity

**Pattern B: CGH in Dedicated Spoke**
```
Hub VNet (ALZ) вҶҗв”Җв”ҖPeeringв”Җв”Җв–ә CGH VNet (Dedicated)
в”ңв”Җв”Җ Firewall              в”ңв”Җв”Җ APIM Subnet
в”ңв”Җв”Җ VPN Gateway           в”ңв”Җв”Җ Monitoring Subnet
в””в”Җв”Җ Shared Services       в””в”Җв”Җ CGH Services
```
- Better separation of concerns
- For: > 100 spokes, large-scale deployments

### Implementation Details

#### VNet Peering Configuration

```bicep
// Hub to Spoke peering
resource hubToSpoke 'Microsoft.Network/virtualNetworks/virtualNetworkPeerings@2023-09-01' = {
  name: '${hubVnetName}/peering-to-${spokeVnetName}'
  properties: {
    allowVirtualNetworkAccess: true
    allowForwardedTraffic: true
    allowGatewayTransit: true    // Hub shares gateway
    useRemoteGateways: false
    remoteVirtualNetwork: {
      id: spokeVnet.id
    }
  }
}

// Spoke to Hub peering
resource spokeToHub 'Microsoft.Network/virtualNetworks/virtualNetworkPeerings@2023-09-01' = {
  name: '${spokeVnetName}/peering-to-${hubVnetName}'
  properties: {
    allowVirtualNetworkAccess: true
    allowForwardedTraffic: false
    allowGatewayTransit: false
    useRemoteGateways: true      // Spoke uses hub gateway
    remoteVirtualNetwork: {
      id: hubVnet.id
    }
  }
}
```

## Consequences

### Positive Consequences

1. **Clear Governance Boundary** - Hub provides single enforcement point
2. **Scalable Architecture** - Add spokes without impacting existing infrastructure
3. **Secure by Default** - Network isolation with controlled connectivity
4. **Enterprise Alignment** - Matches Azure Landing Zone patterns
5. **Cost Clarity** - Per-spoke attribution and shared infrastructure efficiency
6. **Operational Simplicity** - Centralized monitoring and management

### Trade-offs

1. **VNet Peering Complexity** - Managing peering relationships at scale
   - **Mitigation:** Infrastructure-as-code (Bicep/Terraform), automation

2. **Network Latency** - Additional hop through hub
   - **Mitigation:** Peering latency is minimal (< 2ms), regional deployments

3. **Hub Bottleneck** - Central point that could become overloaded
   - **Mitigation:** Auto-scaling, multiple hubs for large deployments

### Validation Checklist

Before deploying hub-spoke topology:

- [ ] IP address space planned and documented
- [ ] Hub capacity assessed (existing vs new VNet)
- [ ] VNet peering limits reviewed (500 per VNet)
- [ ] Firewall rules planned for traffic inspection
- [ ] Private DNS zones configured
- [ ] Gateway transit requirements defined
- [ ] Cost allocation model established

    Identity Management

    Identity and access management patterns for enterprise AI — human, service, and agent identities with Microsoft Entra ID integration

    ​
    🆔 Identity Management

    ​
    Introduction

    Effective identity management is the foundation of AI governance. The Foundry Citadel Platform implements comprehensive identity patterns that distinguish between human identities (developers, operators), service identities (applications, workloads), and agent identities (autonomous AI agents), each with appropriate lifecycle management and access controls.

    Microsoft Entra ID (formerly Azure AD) serves as the central identity provider:
    FeaturePurposeImplementation
    UsersHuman identity managementEmployee accounts with MFA

    Authentication Patterns

    ​
    Multi-Factor Authentication (MFA)

    Required for all human access to AI services: 
    MFA Requirements:
├── Required for: All interactive access
├── Methods: Authenticator app, SMS, hardware key
├── Exemptions: Emergency break-glass accounts only
└── Enforcement: Conditional Access policy

    ​
    Privileged Access Workstations (PAW)

    For high-privilege operations:
    • Dedicated hardened workstations
    • No internet browsing
    • Just-in-time (JIT) access elevation
    • Session recording for audit

    Service Identities

    ​
    App Registrations

    Register applications for programmatic access: 
    # Create app registration
$app = New-AzADApplication `
  -DisplayName "CustomerServiceBot" `
  -SignInAudience "AzureADMyOrg"

# Create service principal
$sp = New-AzADServicePrincipal `
  -ApplicationId $app.AppId

# Assign role
New-AzRoleAssignment `
  -ObjectId $sp.Id `
  -RoleDefinitionName "Cognitive Services User" `
  -Scope "/subscriptions/$subscriptionId/resourceGroups/$rgName"

    Managed Identities

    ​
    System-Assigned Managed Identity

    Automatically created and managed by Azure: 
    {
  "identity": {
    "type": "SystemAssigned"
  }
}
    Use when:
    • Single application per resource
    • Lifecycle tied to Azure resource
    • No credential management needed

    ​
    User-Assigned Managed Identity

    Standalone identity resource: 
    {
  "identity": {
    "type": "UserAssigned",
    "userAssignedIdentities": {
      "/subscriptions/{sub}/resourceGroups/{rg}/providers/Microsoft.ManagedIdentity/userAssignedIdentities/{name}": {}
    }
  }
}
    Use when:
    • Multiple resources share identity
    • Separate lifecycle from resources
    • Cross-resource access patterns

    Service Principal Security

    ​
    Certificate-Based Authentication

    More secure than client secrets: 
    # Generate certificate
$cert = New-SelfSignedCertificate `
  -Subject "CN=AI-Service" `
  -CertStoreLocation "Cert:\CurrentUser\My" `
  -KeyExportPolicy Exportable

# Add certificate to app registration
New-AzADAppCredential `
  -ApplicationId $appId `
  -CertValue $cert.RawData

    Shadow Agent Detection

    Automatically discover unregistered agents:
    Detection MethodData SourceAction
    Traffic AnalysisGateway logsIdentify unknown API keys
    Entra SignalsSign-in logsDetect unrecognized applications
    Network ScanningNSG flow logsFind unauthorized endpoints
    Code ScanningRepositoriesDetect hardcoded credentials
    # Query for potential shadow agents
Get-AzLog `
  -ResourceGroupName "ai-gateway-rg" `
  -StartTime (Get-Date).AddDays(-7) `
  -Caller "Unknown" |
  Where-Object { $_.OperationName -like "*OpenAI*" }

    Authentication Patterns

    ​
    OAuth 2.0 Flows

    ​
    Client Credentials Flow (Service-to-Service)

    For automated service authentication: 
    ┌─────────┐                              ┌─────────────┐          ┌─────────┐
│ Service │                              │    Entra    │          │   AI    │
│    A    │                              │     ID      │          │ Service │
└────┬────┘                              └──────┬──────┘          └────┬────┘
     │                                          │                      │
     │ 1. POST /token                           │                      │
     │    grant_type=client_credentials         │                      │
     │    client_id={app_id}                    │                      │
     │    client_secret={secret}                │                      │
     │ ────────────────────────────────────────>│                      │
     │                                          │                      │
     │ 2. Access Token                          │                      │
     │ <────────────────────────────────────────│                      │
     │                                          │                      │
     │ 3. GET /openai/deployments               │                      │
     │    Authorization: Bearer {token}         │                      │
     │ ────────────────────────────────────────────────────────────────>│
     │                                          │                      │
     │ 4. Response                              │                      │
     │ <────────────────────────────────────────────────────────────────│
    Implementation: 
    // Using Azure.Identity
var credential = new ClientSecretCredential(
    tenantId: "contoso.onmicrosoft.com",
    clientId: "app-id",
    clientSecret: "secret"
);

var token = await credential.GetTokenAsync(
    new TokenRequestContext(new[] { "https://cognitiveservices.azure.com/.default" })
);

    ​
    On-Behalf-Of Flow (Delegated Access)

    For acting on behalf of users: 
    Client → API A → API B (AI Service)

1. Client authenticates with user credentials
2. Client calls API A with user's token
3. API A exchanges token for AI service access
4. API A calls AI service on behalf of user

    // Server-side token exchange
var credential = new OnBehalfOfCredential(
    tenantId: "contoso.onmicrosoft.com",
    clientId: "api-a-app-id",
    clientSecret: "api-a-secret",
    userAssertionToken: incomingUserToken
);

var token = await credential.GetTokenAsync(
    new TokenRequestContext(new[] { "https://cognitiveservices.azure.com/.default" })
);

    Authorization Patterns

    ​
    Role-Based Access Control (RBAC)

    Assign roles to identities: 
    # Assign role to user
New-AzRoleAssignment `
  -SignInName "developer@contoso.com" `
  -RoleDefinitionName "Cognitive Services OpenAI User" `
  -ResourceName "openai-resource" `
  -ResourceGroupName "ai-rg"

# Assign role to service principal
New-AzRoleAssignment `
  -ObjectId $servicePrincipalId `
  -RoleDefinitionName "Cognitive Services OpenAI Contributor" `
  -Scope "/subscriptions/$subId/resourceGroups/$rgName/providers/Microsoft.CognitiveServices/accounts/$accountName"

    ​
    Attribute-Based Access Control (ABAC)

    Fine-grained conditions: 
    {
  "condition": "(
    (
      !(ActionMatches{'Microsoft.CognitiveServices/accounts/deployments/read'})
    )
    OR
    (
      @Resource[Microsoft.CognitiveServices/accounts/deployments:model.name] StringEquals 'gpt-35-turbo'
    )
  )",
  "conditionVersion": "2.0"
}

    Custom Authorization Policies

    APIM policy for custom authorization: 
    <policies>
  <inbound>
    <!-- Check custom claims -->
    <choose>
      <when condition="@(!context.User.Groups.Any(g => g.Name == "AI-Users"))">
        <return-response>
          <set-status code="403" reason="Forbidden" />
          <set-body>{
            "error": "User not authorized for AI access. Contact your administrator to join the AI-Users group."
          }</set-body>
        </return-response>
      </when>
    </choose>
    
    <!-- Check environment access -->
    <set-variable name="user-env" value="@(context.User.Groups.FirstOrDefault(g => g.Name.StartsWith("Env-"))?.Name ?? "none")" />
    <choose>
      <when condition="@(!context.Request.Url.Path.Contains(context.Variables.GetValueOrDefault("user-env", "").Replace("Env-", "").ToLower()))">
        <return-response>
          <set-status code="403" reason="Forbidden" />
          <set-body>{
            "error": "User not authorized for this environment"
          }</set-body>
        </return-response>
      </when>
    </choose>
  </inbound>
</policies>

    Agent Sponsorship Model

    Enforce human accountability: 
    <policies>
  <inbound>
    <!-- Validate agent has active sponsor -->
    <send-request mode="new" response-variable-name="sponsor-check">
      <set-url>{{agent-365-url}}/agents/{{contract.agentId}}/sponsor</set-url>
      <set-method>GET</set-method>
    </send-request>
    
    <choose>
      <when condition="@(((IResponse)context.Variables["sponsor-check"]).StatusCode != 200)">
        <return-response>
          <set-status code="403" reason="Agent Not Authorized" />
          <set-body>{
            "error": "Agent does not have an active sponsor. Contact your platform team."
          }</set-body>
        </return-response>
      </when>
    </choose>
    
    <!-- Add sponsor info to headers for audit -->
    <set-header name="x-agent-sponsor" exists-action="override">
      <value>@(((IResponse)context.Variables["sponsor-check"]).Body.As<JObject>()["sponsor"]?.ToString())</value>
    </set-header>
  </inbound>
</policies>

    Access Reviews

    Periodic certification of access: 
    Access Review Schedule:
├── AI Administrators: Monthly
├── AI Contributors: Quarterly
├── AI Users: Semi-annually
├── Service Principals: Quarterly
├── Agent Sponsors: Quarterly
└── Emergency Access: After each use
    Configuration: 
    # Create access review
New-MgIdentityGovernanceAccessReviewDefinition `
  -DisplayName "AI Service Access Review" `
  -Description "Review access to AI services" `
  -Scope ({
    "@odata.type": "#microsoft.graph.groupAccessReviewScope",
    "groupId": "ai-users-group-id"
  }) `
  -Reviewers ({
    "@odata.type": "#microsoft.graph.groupOwnersReviewers",
    "groupId": "ai-admins-group-id"
  }) `
  -Settings ({
    "recurrence": {
      "pattern": { "type": "quarterly" },
      "range": { "type": "noEnd" }
    },
    "defaultDecision": "Deny",
    "autoApplyDecisionsEnabled": true
  })

    Token Exchange Patterns

    Exchange tokens across boundaries: 
    // Token exchange for cross-spoke access
var request = new HttpRequestMessage(HttpMethod.Post, "https://login.microsoftonline.com/{tenant}/oauth2/v2.0/token");
request.Content = new FormUrlEncodedContent(new Dictionary<string, string>
{
    ["grant_type"] = "urn:ietf:params:oauth:grant-type:jwt-bearer",
    ["client_id"] = "{spoke-b-app-id}",
    ["client_secret"] = "{secret}",
    ["assertion"] = "{token-from-spoke-a}",
    ["scope"] = "https://cognitiveservices.azure.com/.default",
    ["requested_token_use"] = "on_behalf_of"
});

    Best Practices

    ​
    Identity Hygiene

    1. Regular Reviews: Quarterly access reviews for all AI identities
    2. Least Privilege: Start with minimal permissions, expand as needed
    3. Time-Bound Access: Set expiration dates on all non-permanent access
    4. Credential Rotation: Automate secret and certificate rotation
    5. Monitoring: Alert on anomalous access patterns

    ​
    Agent Identity Management

    1. One Identity Per Agent: Never share agent identities
    2. Sponsor Verification: Validate sponsor before agent activation
    3. Lifecycle Integration: Disable agent identity when agent retired
    4. Shadow Detection: Regular scans for unregistered agents
    5. Access Packages: Use pre-approved permission bundles

    ​
    Security Considerations

    1. No Hardcoded Secrets: Use Managed Identities or Key Vault
    2. Network Isolation: Private endpoints for identity traffic
    3. Audit Everything: Log all identity operations
    4. Break-Glass Ready: Emergency access procedures documented
    5. Regular Testing: Penetration testing of identity flows

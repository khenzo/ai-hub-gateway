1.1 · Foundation

The Foundation: Azure Landing Zone

Enterprise-scale infrastructure substrate for AI workloads — before a single agent is built.

A Landing Zone is not a server, not a subscription, and not a template. It is a pre-engineered piece of Azure — a vetted composition of identity, networking, security, monitoring, and policy that an organisation can drop a workload into and have it inherit enterprise standards by default.

Think of it as the difference between buying a plot of land and buying a serviced industrial estate. Both let you build a factory. Only one gives you power, water, sewerage, fibre, fencing, fire suppression and a postal code on day one.

For AI, that distinction is decisive. Without a Landing Zone, every team building an agent must also build the network controls, the identity model, the secrets management, and the audit pipeline. With one, they inherit those things and focus on the agent.

What you get on day one
  • Network isolation — hub-spoke VNets, private endpoints, Azure Firewall, NSGs.
  • Identity backbone — Microsoft Entra ID, RBAC, managed identities, conditional access.
  • Operational baseline — central monitoring, logging, backup, policy enforcement.
  • Compliance posture — Azure Policy, Defender for Cloud, Purview wired in.
  • Cost discipline — tagging, budgets, chargeback ready to use.

2.2 · Why a hub at all?

Without a hub, every team rebuilds the same five things badly.

Without Citadel
  • Bring-your-own auth. Every team integrates Entra differently — or worse, ships with API keys.
  • No rate limit. A runaway notebook can burn six figures of token budget overnight.
  • No content safety. Prompt-injection and jailbreak vectors hit the model directly.
  • No PII masking. Customer data leaves the perimeter the moment a prompt is sent.
  • No audit. When legal asks “what did the model see?”, no one can answer.
With Citadel
  • One auth model. Every caller is an Entra principal with a managed identity, end-to-end.
  • Token & cost limits. APIM enforces per-team, per-app, per-model quotas, with chargeback.
  • Content safety inline. Azure AI Content Safety + jailbreak detection on the request path.
  • PII masking on-ramp / off-ramp. Sensitive entities are redacted and re-hydrated server-side.
  • Complete audit. Who, what, when, where, why, how — for seven years.

The hub is the place where these five concerns are solved once, then inherited by every spoke, every framework, every agent.

The unified AI gateway.

A single Azure API Management front door that every AI request — from every framework, every agent, every notebook, every Power Platform flow — is required to traverse.

The gateway is not a proxy that adds a header. It is a policy-rich request fabric. Each call enters with a JWT, leaves with a redacted response, and along the way is rate-limited, content-checked, PII-masked, smartly routed to the cheapest acceptable model, and written to an immutable log.

Crucially, it is unbypassable. Network policy on the spokes denies egress to Azure OpenAI, Foundry, OpenAI public endpoints, and Anthropic APIs — the only route out of the platform is through APIM.

2.5 · Separation of concerns, unified at runtime

Layers in motion.

Each layer owns a single concern. None of them can do the others’ jobs. That separation is what makes the architecture auditable: every property of the platform can be traced to exactly one layer.

Layer 1 · Gateway

The only door.

APIM, the policies above, smart routing, fallback, exit points to model providers. If a request is on the platform, it is on this door — there is no other way in or out.

Layer 2 · AI Control Plane

Policy & observability.

Model allow-lists, residency rules, evaluation gates, prompt registries, lineage capture, cost attribution. The control plane is where governance lives as code — versioned, reviewed, deployed.

Layer 3 · Agent Identity

Every agent is a principal.

No service principals with secrets. No PATs. No shared API keys. Each agent and each tool is registered in Entra and acts under its own managed identity, with on-behalf-of flows for user impersonation.

Layer 4 · Security Fabric

Wraps everything else.

Private endpoints, network isolation, Key Vault, customer-managed keys, Defender for Cloud, Purview classification. The fabric is invisible from the request path but present at every boundary.

The flow runs L4 → L3 → L2 → L1 → agent. A change in any layer cannot weaken the layer below it — that is the platform-enforced boundary.

3.1 · Three guarantees the data layer must make

Three non-negotiable guarantees for governed AI data.

An AI agent that asks “what is ARR by region for Q1?” must satisfy three properties before a single byte of data is returned. Without them, you have a chatbot wired to a warehouse — not a governed enterprise capability.

Governed identity at every hop
No service principals with secrets. No PATs in notebooks. The agent acts as the user, on-behalf-of, end-to-end.
Governance by construction
Row filters and column masks are part of the metric definition itself — not a wrapper that a clever query can dodge.
Full observability
Every query — agent or human — produces lineage. From prompt to SQL to row, the path is recorded and replayable.
20 / 46
3.2 · Why most AI fails at the data layer

The model is rarely the bottleneck.
The data plumbing is.

A successful AI demo runs against a curated dataset, a privileged notebook, and a user who happens to have access to everything. None of those conditions hold in production.

In production, the same prompt has to work for a junior analyst in EMEA, a privileged finance director in APAC, and an external auditor with read-only scope — and produce different, governed answers for each. That is not a model problem. It is a plumbing problem.

Common failure modes
  • Service principal with broad scope. The agent reads everything because the SP can.
  • Hardcoded SQL. Two agents asking the same question produce different numbers.
  • RAG over raw PDFs. No row-level security, no lineage, no recall when a doc is retracted.
  • Prompt leakage. PII passes to the model because nothing inspected the prompt.
  • No certification path. Local team-built metrics conflict with enterprise definitions.

Each of these has a known fix. Putting them together as one architecture — that is the Data Foundation chapter.

3.4 · Identity-aware access

Same prompt. Same agent. Different data.

3.5 · The semantic layer

Author once.
Reuse everywhere.

A metric — ARR Run-Rate, say — is defined exactly once, in exactly one place, with exactly one formula, exactly one set of dimensions, and exactly one row-filter rule.

Power BI dashboards consume that definition. Jupyter notebooks consume that definition. AI agents consume that definition. They cannot drift from it because there is no other source.

This is implemented natively on the Microsoft and Databricks stack: MetricFlow (semantic models) plus Unity Catalog (governance & access) plus OneLake (storage) — no third-party semantic layer required.

4.5 · The agent lifecycle

Born governed. Retired governed.

An agent on Citadel has a documented lifecycle — the same way a Kubernetes workload or an APIM API does. There is no ‘just put it in production’ path.

1 · Register

Agent is created in the registry with owner, purpose, allowed models, allowed tools, allowed data sources, evaluation suite, and HITL policy.

2 · Evaluate

Pre-deploy evals on a fixed dataset. Quality, safety, refusal-rate, PII-leakage, jailbreak resistance — all gated. No green eval, no deploy.

3 · Deploy

Identity provisioned, gateway route created, quota assigned, audit binding wired up. The agent gets exactly the scope its registry entry asks for — nothing more.

4 · Observe

Live evals on real traffic, drift detection, cost dashboards, refusal monitoring, prompt & response sampling for offline review.

5 · Iterate

Prompt changes, tool changes, model changes — each is a versioned deployment, each re-runs evals, each is reversible by rollback.

6 · Retire

Identity revoked, route removed, audit data preserved per retention schedule. The retired agent leaves a complete forensic record behind.

5.3 · The contract between hub and spoke

A handshake in code.

‘Contract’ is not a metaphor. It is a versioned manifest, signed at deploy time, that declares what a spoke is allowed to do and what the hub guarantees in return. If either side breaks it, the route fails closed.

Spoke declares
  • Identity. The Entra principal(s) that will originate AI calls.
  • Use cases. Named applications — e.g. finance-arr-agent.
  • Models requested. From the central allow-list, with rate limits.
  • Data scopes. Which Unity Catalog assets the spoke needs to read.
  • Cost ceiling. Monthly token budget, with kill-switch behaviour.
Hub guarantees
  • Gateway availability. APIM SLO + multi-region failover.
  • Policy stability. Versioned policies; breaking changes announced.
  • Audit retention. 7 years immutable, with legal hold support.
  • Cost transparency. Daily chargeback feed with line-item detail.
  • Incident SLA. Defined response times for security & availability events.

Risks & mitigations

Where this goes wrong — and how we stop it.

An honest deck names the failure modes. None of these are hypothetical — each one has a documented countermeasure already wired into the platform.

Risk · gateway becomes bottleneck

A single APIM region can become the failure domain for the whole platform.

Mitigation: multi-region active-active APIM with geo-routing; private link to model providers in each region; documented RTO/RPO.

Risk · cost runaway

A misbehaving agent loops, exhausting the spoke’s monthly token budget in hours.

Mitigation: hard quota at gateway with soft + hard alerts; per-conversation circuit breakers; daily chargeback feed; default kill-switch behaviour.

Risk · semantic drift

Edge metrics quietly diverge from Core; two reports cite different ARR.

Mitigation: name-collision rejection at deploy; periodic Core/Edge reconciliation; promotion path made the path of least resistance.

Risk · framework lock-in

A spoke standardises on a framework that the vendor later abandons.

Mitigation: tool-and-context interfaces are framework-neutral; agent registry holds the contract, not the framework; migration is a re-skin.

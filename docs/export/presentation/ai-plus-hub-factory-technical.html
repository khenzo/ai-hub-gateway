1.1 · Foundation

Infrastructure Substrate: Azure Landing Zone

Pre-engineered Azure composition for AI workloads — identity, networking, security, monitoring, and policy inherited by default.

A Landing Zone is not a server or subscription — it is a vetted composition of Azure services that provides enterprise standards by default. For AI workloads, this means every agent inherits network controls, identity model, secrets management, and audit pipelines without rebuilding them.

The AI+ FACTORY Hub extends Microsoft's Cloud Adoption Framework (CAF) Landing Zones with AI-specific design areas: private endpoints for Azure OpenAI, VNet isolation for model endpoints, and managed identity patterns for agent auth.

1.3 · Network architecture

Hub-spoke topology with private endpoints.

All AI traffic flows through the hub. Spokes cannot reach model endpoints directly. Network policy enforces the contract — not documentation.

Hub network
  • Azure Firewall — centralised egress/ingress inspection
  • Private DNS zones — privatelink.openai.azure.com
  • VPN/ExpressRoute gateway — hybrid connectivity
  • DDoS Protection Standard — perimeter defence
  • Application Gateway + WAF — web workload protection
Spoke network
  • VNet peering — transitive through hub, no direct spoke-to-spoke
  • NSG rules — deny all outbound except hub and Microsoft services
  • Private endpoints — Azure OpenAI, Storage, Key Vault, SQL

2.2 · Why a hub at all?

Without a hub, every team rebuilds the same controls differently.

Without Citadel
  • Bring-your-own auth. Teams integrate Entra differently — or ship with API keys in notebooks.
  • No rate limits. A runaway loop burns token budgets. No quota enforcement at the edge.
  • No content safety. Prompt-injection and jailbreak vectors hit the model directly.
  • No PII masking. Customer data leaves the perimeter before anyone inspects the prompt.
  • No audit trail. When asked "what did the model see?", no structured answer exists.
With Citadel
  • One auth model. Every caller is an Entra principal with managed identity, end-to-end.
  • Token & cost quotas. APIM enforces per-subscription QPM/TPM limits with chargeback.
  • Content safety inline. Azure AI Content Safety + jailbreak detection on the request path.
  • PII masking on-ramp/off-ramp. Sensitive entities redacted and re-hydrated server-side.
  • Complete audit. Who, what, when, where, why, how — structured events to Event Hub.

The hub solves these concerns once, then inherits them by every spoke, every framework, every agent.

2.3 · Layer 1 · The Gateway

Unified AI gateway.

Single Azure API Management front door. Every AI request — from every framework, agent, notebook, Power Platform flow — must traverse it.

The gateway is a policy-rich request fabric. Each call enters with a JWT, leaves with a redacted response, and along the way is rate-limited, content-checked, PII-masked, routed to the cheapest acceptable model, and written to an immutable log.

It is unbypassable. Network policy on spokes denies egress to Azure OpenAI, Foundry, OpenAI public endpoints, and Anthropic APIs. The only route out is through APIM.

2.5 · Separation of concerns

Layers in motion.

Each layer owns a single concern. That separation makes the architecture auditable: every property traces to exactly one layer.

Layer 1 · Gateway

The only door.

APIM, policies, smart routing, fallback, exit points to model providers. If a request is on the platform, it is on this door — no other way in or out.

Layer 2 · AI Control Plane

Policy & observability.

Model allow-lists, residency rules, evaluation gates, prompt registries, lineage capture, cost attribution. Governance lives as code — versioned, reviewed, deployed.

Layer 3 · Agent Identity

Every agent is a principal.

No service principals with secrets. No PATs. No shared API keys. Each agent and tool is registered in Entra and acts under its own managed identity, with on-behalf-of flows for user impersonation.

Layer 4 · Security Fabric

Wraps everything else.

Private endpoints, network isolation, Key Vault, customer-managed keys, Defender for Cloud, Purview classification. Invisible from the request path but present at every boundary.

Flow runs L4 → L3 → L2 → L1 → agent. A change in any layer cannot weaken the layer below it — platform-enforced boundary.

3.1 · Three guarantees the data layer must make

Three non-negotiable guarantees.

Before a single byte returns, the data layer must satisfy three properties. Without them, you have a chatbot wired to a warehouse — not a governed capability.

Guarantee 1

Governed identity at every hop

No service principals with secrets. No PATs in notebooks. The agent acts as the user, on-behalf-of, end-to-end. Entra identity chains from gateway to Unity Catalog to OneLake.

Guarantee 2

Governance by construction

Row filters and column masks are SQL UDFs attached to table definitions. They execute inside the query engine — masked data is never materialized. No wrapper a clever query can dodge.

Guarantee 3

Full observability

Every query — agent or human — produces lineage. From prompt to SQL to row, the path is recorded in Unity Catalog system tables and streamed to Event Hub.

21 / 47
3.2 · Why most AI fails at the data layer

The model is rarely the bottleneck.
The data plumbing is.

A successful AI demo runs against a curated dataset, a privileged notebook, and a user with access to everything. None of those conditions hold in production.

In production, the same prompt works for a junior analyst in EMEA, a finance director in APAC, and an external auditor — producing different, governed answers for each. That is not a model problem. It is a plumbing problem.

Common failure modes
  • Service principal with broad scope. The agent reads everything because the SP can.
  • Hardcoded SQL. Two agents asking the same question produce different numbers.
  • RAG over raw PDFs. No row-level security, no lineage, no recall when retracted.
  • Prompt leakage. PII passes to the model because nothing inspected the prompt.
  • No certification path. Local team-built metrics conflict with enterprise definitions.

Each of these has a known fix. Putting them together as one architecture — that is the Data Foundation chapter.

3.4 · Identity-aware access

Same prompt. Same agent. Different data.

The same question, asked by three different users, returns three different governed answers. The agent does not decide who sees what — Unity Catalog does, on the basis of the original Entra identity.

Junior analyst · EMEA
SELECT ...
WHERE region = 'EMEA'
AND mask(customer_name)
Sees aggregated EMEA only. Customer names masked.
Finance director · APAC
SELECT ...
WHERE region IN ('APAC')
-- no mask
Sees full APAC detail, including PII, with audit.
External auditor
SELECT ...
WHERE in_audit_scope = 1
AND mask(*)
Sees in-scope rows only. All PII masked.

The agent did not write any of these WHERE clauses. Unity Catalog injected them based on the verified Entra principal that arrived at the gateway.

3.5 · The semantic layer

Author onc.
Reuse everywhere.

A metric — ARR Run-Rate, say — is defined exactly once, in exactly one place, with exactly one formula, one set of dimensions, and one row-filter rule.

Power BI dashboards consume that definition. Jupyter notebooks consume that definition. AI agents consume that definition. They cannot drift from it because there is no other source.

Implemented natively on the Microsoft + Databricks stack: MetricFlow (semantic models) plus Unity Catalog (governance & access) plus OneLake (storage) — no third-party semantic layer required.

4.5 · The agent lifecycle

Born governed. Retired governed.

An agent on the platform has a documented lifecycle — the same way a Kubernetes workload or an APIM API does. There is no 'just put it in production' path.

1 · Register

Agent is created in the registry with owner, purpose, allowed models, allowed tools, allowed data sources, evaluation suite, and HITL policy.

2 · Evaluate

Pre-deploy evals on a fixed dataset. Quality, safety, refusal-rate, PII-leakage, jailbreak resistance — all gated. No green eval, no deploy.

3 · Deploy

Identity provisioned, gateway route created, quota assigned, audit binding wired up. The agent gets exactly the scope its registry entry asks for — nothing more.

4 · Observe

Live evals on real traffic, drift detection, cost dashboards, refusal monitoring, prompt & response sampling for offline review.

5 · Iterate

Prompt changes, tool changes, model changes — each is a versioned deployment, each re-runs evals, each is reversible by rollback.

6 · Retire

Identity revoked, route removed, audit data preserved per retention schedule. The retired agent leaves a complete forensic record behind.

5.1 · One hub · many spokes

Govern central.

5.3 · The contract between hub and spoke

A handshake in code.

'Contract' is not a metaphor. It is a versioned manifest, signed at deploy time, that declares what a spoke is allowed to do and what the hub guarantees in return. If either side breaks it, the route fails closed.

Spoke declares
  • Identity. The Entra principal(s) that will originate AI calls.
  • Use cases. Named applications — e.g. finance-arr-agent.
  • Models requested. From the central allow-list, with rate limits.
  • Data scopes. Which Unity Catalog assets the spoke needs to read.
  • Cost ceiling. Monthly token budget, with kill-switch behaviour.
Hub guarantees
  • Gateway availability. APIM SLO + multi-region failover.
  • Policy stability. Versioned policies; breaking changes announced.
  • Audit retention. 7 years immutable, with legal hold support.
  • Cost transparency. Daily chargeback feed with line-item detail.
  • Incident SLA. Defined response times for security & availability events.

Risks & mitigations

Where this goes wrong — and how we stop it.

An honest deck names the failure modes. None of these are hypothetical — each one has a documented countermeasure already wired into the platform.

Risk · gateway becomes bottleneck

A single APIM region can become the failure domain for the whole platform.

Mitigation: multi-region active-active APIM with geo-routing; private link to model providers in each region; documented RTO/RPO.

Risk · cost runaway

A misbehaving agent loops, exhausting the spoke's monthly token budget in hours.

Mitigation: hard quota at gateway with soft + hard alerts; per-conversation circuit breakers; daily chargeback feed; default kill-switch behaviour.

Risk · semantic drift

Edge metrics quietly diverge from Core; two reports cite different ARR.

Mitigation: name-collision rejection at deploy; periodic Core/Edge reconciliation; promotion path made the path of least resistance.

Risk · framework lock-in

A spoke standardises on a framework that the vendor later abandons.

Mitigation: tool-and-context interfaces are framework-neutral; agent registry holds the contract, not the framework; migration is a re-skin.

    Data Foundation

    Semantic Layer

    Why platform-native semantics (MetricFlow + Unity Catalog Business Semantics) are required for production AI agents — the author-once principle and the Core vs Edge governance model.

    The central problem with tool-embedded business logic is inheritance: when metric definitions live inside BI tools — DAX in Power BI, LookML in Looker, MDX in SSAS — AI agents inherit nothing. To answer a business question, an agent must improvise business definitions from raw table schemas, producing results that are plausible, internally consistent, and entirely ungoverned. Two agents asking the same question may produce different SQL and return different numbers. Neither agent is wrong by its own reasoning; both are wrong by the organization’s definition. When semantics live in the data platform next to the data and governance policies, every consumer — dashboards, notebooks, and AI agents — reads from the same governed truth. The semantic layer is the infrastructure that determines whether AI-driven analytics is trustworthy or dangerously plausible. It is not a reporting convenience. It is the mechanism that closes the gap between what an AI model produces and what the business defines. This page covers:
    • The decision between platform-native and tool-embedded semantics — an opinionated recommendation, not a neutral comparison
    • MetricFlow + UC Business Semantics: How these two layers complement each other (transformation time vs consumption time) and why they are not competing alternatives
    • AI Agent Grounding: How production AI agents interact with the semantic layer in exactly two phases — grounding and execution
    • Author Once, Reuse Everywhere: The metric lifecycle traced from authorship through Unity Catalog Business Semantics to identical consumption by dashboards, notebooks, and AI agents
    • Core vs Edge Governance: What belongs in the Core certified model, what belongs in Edge team models, and how Edge metrics are promoted to Core
    The Semantic Layer is the grounded intelligence substrate of the Enterprise Data Foundation stack. It connects to the AI+ / Citadel 4-layer model at Layer 1 (APIM governs semantic endpoint access) and inherits governance from Unity Catalog (column masks and row filters travel with metric definitions). It is not an additional architectural layer — it is the mechanism that makes Unity Catalog governance actionable for AI consumers.

    ​
    Platform-Native vs Tool-Embedded Semantics

    ​
    Decision Question

    Where should business metric definitions live — inside BI tools or inside the data platform?

    ​
    The Problem with Tool-Embedded Semantics

    Tool-embedded semantics force organizations to re-implement governance in every tool. When an organization adds a new BI tool, all metric definitions must be re-implemented in that tool’s proprietary language. When Power BI calculates ARR Run-Rate one way and Looker calculates it another, both calculations may be defensible — but neither is the business definition. The inconsistency is not a bug in any single tool; it is a structural property of embedding semantics inside tools rather than in the platform. The AI agent problem is more severe. When an agent needs to answer a business question, it has access to table schemas and column names. It does not have access to the LookML or DAX definitions that encode business logic, calculation rules, dimension hierarchies, and access constraints. The agent must infer ARR Run-Rate from SUM(mrr) * 12 / active_count — if the column names are lucky enough to be descriptive. If the agent guesses wrong, the result is incorrect. If the agent guesses inconsistently across sessions, the analytics are unauditable.
    “The biggest risk isn’t a wrong number. It’s a system where the right number depends on where you ask the question.”
    Tool-embedded semantics create exactly this condition. The right answer for ARR Run-Rate depends on whether you ask Power BI, Looker, or an AI agent querying the raw tables. The inconsistency is silent and systemic — most organizations discover it only when two reports disagree in a board presentation.

    ​
    Recommended Answer

    Platform-native semantics via MetricFlow + Unity Catalog Business Semantics are required for enterprise AI deployments.
    “For enterprise analytics, where consistency and auditability are non-negotiable, the semantic layer is not optional. It is the architecture that makes AI-driven analytics trustworthy.”
    This recommendation assumes Unity Catalog Business Semantics metric views are defined and certified. If that layer does not exist, text-to-SQL against governed tables with schema documentation is the interim path — with full acknowledgment of its governance limitations. Text-to-SQL is acceptable for exploratory analysis, developer prototyping, and human-validated use cases where consistency requirements are relaxed. It is not a production-grade substitute for enterprise AI agents where auditability is required. The distinction is not about capability; it is about governance. Text-to-SQL agents cannot demonstrate that their output is derived from a certified business definition. Platform-native semantic agents can, because the certified definition travels with every query.

    ​
    MetricFlow and Unity Catalog Business Semantics: Complementary Layers

    MetricFlow (dbt) and Unity Catalog Business Semantics are not competing alternatives. They address different layers in the semantic stack and serve distinct functions in the metric lifecycle. Treating them as alternatives is a category error — it is equivalent to comparing a compiler to a runtime.

    MetricFlow metric definitions feed into Unity Catalog Business Semantics as the platform-native exposure layer. The Unity Catalog ABAC policies, column masks, and row filters documented in Unity Catalog Governance travel with every metric view served through this layer. MetricFlow ensures the definition is correct and tested; Unity Catalog Business Semantics ensures it is governed and accessible to every consumer through open APIs. Five implementation principles govern the platform-native semantic layer approach:
    1. Author once, reuse everywhere — definitions are platform-native assets, not embedded in charts or reports
    2. Proximity to governance — access controls, traceability, and certification travel with the asset, not the consuming surface
    3. Openness by design — prefer REST, GraphQL, JDBC over proprietary DSL lock-in; any tool can consume the same governed definition
    4. One source for humans and AI — the same metric definitions serve BI dashboards and conversational AI agents; there is no separate AI-optimized layer
    5. Semantics as code — definitions are versioned, reviewed, and deployed through CI/CD pipelines; promotion from dev to staging to prod is a controlled workflow

    ​
    AI Agent Grounding: How Agents Query the Semantic Layer

    For production AI agents, semantic layer grounding is required, not optional. Text-to-SQL is acceptable only where consistency and auditability requirements are relaxed. The architectural distinction is precise: grounded agents query governed metric definitions; text-to-SQL agents query raw table schemas. The difference in governance posture between these two approaches is not incremental — it is categorical.

    ​
    The Two-Phase Agent Interaction

    AI agents interact with the semantic layer in exactly two phases: Phase 1 — Grounding (before any query is generated) The agent reads semantic layer descriptive context: available metrics, dimension definitions, governance rules, synonym mappings, and business glossary terms. This phase prevents hallucinated column names, incorrect joins, and misapplied filters. Before generating any SQL, the agent understands the governed vocabulary — it knows that “ARR” maps to the ARR Run-Rate metric definition, that the definition uses SUM(mrr) * 12 / active_count, and that the row filter region = current_user_region() will be applied. The agent does not invent these definitions. It reads them. Phase 2 — Execution (governed query, not raw table scan) The agent queries governed metric definitions, not raw base tables. Measures, dimensions, filters, and access controls travel with the metric definition. The output is consistent across all consumers because the same metric definition — with the same formula, dimensions, and access controls — serves every surface. A BI dashboard, a Jupyter notebook, and a conversational AI agent all produce identical output for the same metric request, because they all query the same governed definition.

    ​
    Why Text-to-SQL Fails at Enterprise Scale

    Text-to-SQL generates SQL from schema metadata only. The LLM has no access to business definitions, governance rules, or certified calculation logic. It must infer intent from column names and table relationships. The results share three structural properties that make them unsuitable for production enterprise analytics:
    • Inconsistent: two agents querying the same business question produce different SQL because they are generating rather than reading the calculation logic
    • Ungoverned: row-level filters and column masks are not automatically applied — the agent queries raw tables that may return data the caller should not see
    • Unauditable: there is no lineage from agent query back to a certified business definition; the organization cannot demonstrate that an agent’s output is derived from an approved calculation
    These are not failure modes that can be addressed by improving the LLM. They are structural properties of an architecture that places business logic generation inside the agent rather than in the platform.

    ​
    Genie as the Semantic Layer Interface

    Genie (Databricks AI/BI) is the consumption interface for Unity Catalog Business Semantics. It reads metric definitions, synonyms, and guardrails from the semantic layer before generating any query, and routes execution through governed metric views rather than raw table SQL. A Genie conversation that asks “What is our ARR by region for Q1?” resolves “ARR” to the certified ARR Run-Rate metric definition, applies the caller’s row filter, and returns governed output — without the caller specifying table names, joins, or filter logic.
    Genie file uploads are in Public Preview. Core Genie features (conversational analytics against Unity Catalog semantic models) are GA. Do not rely on file upload capabilities for production AI agent grounding workflows.

    ​
    The Architectural Guarantee

    When an AI agent queries through the semantic layer, every response is grounded in the same business logic as the BI dashboard and the dbt test. The semantic layer closes the gap between “what the model produces” and “what the business defines.” This guarantee is architectural, not operational — it holds because the definition lives in the platform, not because the organization has built processes to check consistency after the fact.

    ​
    Author Once, Reuse Everywhere

    A metric definition is a platform-native asset. It lives in one place — Unity Catalog Business Semantics — and serves every consumer with identical governed output. The author-once principle is not a documentation convention; it is an architectural property of the platform-native semantic layer. A definition that lives in Unity Catalog cannot diverge across consumers, because all consumers read from the same source.

    ​
    Tracing a Single Metric Definition

    A finance analyst defines ARR Run-Rate in the Unity Catalog Business Semantics layer: formula, standard dimensions (Customer Segment, Region, Fiscal Period), applicable row filter (region = current_user_region()), and synonym mappings (“annual recurring revenue”, “ARR”, “run rate”). This single definition is authored once. From that moment, every consumer — BI dashboards, Jupyter notebooks, and AI agents — reads from the same governed source. The row filter enforces access control for every consumer identically. A Power BI dashboard, a dbt notebook, and a conversational AI agent all calculate ARR Run-Rate the same way. No re-implementation. No divergence.

    ​
    Governance Travels with the Definition

    Row-level security and column masking are not applied at the BI tool or the agent layer — they are applied by Unity Catalog when any consumer queries the metric definition. The security boundary is the semantic layer, not the consuming surface. This is governance by construction, not governance by convention. The distinction matters operationally. Governance by convention relies on every tool and every team correctly implementing the same access controls. When a new tool is added — a new BI platform, a new AI agent framework, a new notebook environment — the convention must be re-applied. Governance by construction requires no re-application. The metric definition carries its access controls into every new consumption context automatically.

    ​
    Impact Analysis Before Change

    When a metric definition needs to change — formula update, dimension added, filter modified — Unity Catalog lineage shows every dashboard, notebook, and AI agent that depends on that definition before the change is committed. No surprise downstream breakage. This is the operational value of semantics as a platform-native asset. The organization does not discover that a formula change broke a board-level dashboard at the next board presentation; it discovers it during the review of the lineage impact analysis, before the change is deployed.

    ​
    Core vs Edge Semantic Model Governance

    Enterprise semantic models serve two fundamentally different requirement profiles simultaneously. Enterprise-wide reporting requirements are stable, certified, and slow-changing — the definition of ARR Run-Rate should not change between fiscal quarters. Team-specific, application-specific, and agent-specific requirements are fast-moving and experimental — a sales operations team may need a pipeline conversion rate definition that is specific to their funnel model. A flat governance model applied to both requirement profiles produces either a bureaucratic bottleneck (if enterprise standards apply to every new metric) or semantic drift (if teams maintain separate definitions that gradually diverge from the enterprise standard). The Core vs Edge model resolves this tension by separating governance tiers.

    ​
    Core: Certified Enterprise Metrics

    The Core semantic model contains:
    • Authoritative metric definitions certified at the enterprise level
    • Standard dimensions: Customer Segment, Region, Fiscal Period
    • Enterprise-wide row-level security and column masking policies applied to all consumers
    • Formal change review required — Unity Catalog lineage impact analysis must be run before any definition change is committed
    • Representative metrics: ARR Run-Rate, NRR, Churn Rate, CAC, EBITDA
    Core metrics change slowly by design. Correct rather than fast is the governing constraint. A Core metric that changes frequently is a signal of inadequate initial definition, not a signal that the governance process is too slow. The stability of Core metrics is what allows downstream consumers — BI dashboards, agent workflows, compliance reports — to depend on them without versioning risk.

    ​
    Edge: Team-Specific Enrichment

    Edge semantic models are per-team, per-application, or per-agent semantic models seeded from Core:
    • Inherits all Core definitions and governance — Core metrics are available as read-only dependencies
    • Enhanced with local synonyms, domain-specific filters, experimental metrics, and agent-specific guardrails
    • Team-owned and team-managed — no central review gate for new Edge additions
    • Representative metrics by domain: Pipeline Conversion Rate (Sales Edge), FX-Adjusted ARR (Finance Edge), Churn Risk Propensity Score (AI Agent Edge)
    Edge models allow teams to move fast without breaking the enterprise semantic model. A finance team can define an FX-adjusted ARR variant without modifying the Core ARR Run-Rate definition. A sales team can add synonyms for pipeline metrics that match their internal vocabulary. An AI agent team can add guardrails and exclusion rules specific to a conversational agent’s scope — without those constraints affecting the BI dashboard consuming the same Core metric.

    ​
    The Promotion Path

    Edge metrics that prove useful and stable must have a defined path to Core. Without a formal promotion path, successful Edge metrics proliferate as parallel definitions — teams reference the Edge version, it becomes the de facto standard, and the Core definition atrophies. The promotion path prevents semantic drift while allowing innovation at the Edge. Promotion criteria for Edge to Core:
    1. The metric has been in production for a defined stabilization period (team-defined, typically one to two quarters)
    2. The definition passes Core review against formula standards, dimension alignment, and naming conventions
    3. A Unity Catalog lineage impact analysis is run to identify all Edge dependents before promotion
    4. The promoted metric replaces the Edge version — consumers migrate to the Core definition with lineage preserved
    The promotion path is an architectural requirement, not a governance convenience. Without it, the Core model stagnates while Edge models accumulate the organization’s actual working definitions. Over time, the “Core” label becomes a historical artifact rather than a guarantee of currency.

    ​
    Governance Enforcement

    Core definitions cannot be overridden by Edge consumers. An Edge team can extend a Core metric with local filters and synonyms, but cannot alter the Core formula. This constraint is enforced by Unity Catalog permissions — Core metric views are owned by a governed service account; Edge teams have read-only access to Core definitions. The permission boundary is not a process guardrail; it is a platform enforcement point.

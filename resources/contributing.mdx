## Reporting Issues

### Documentation Issues

If you find errors, unclear explanations, or missing information:

1. Check if the issue already exists in GitHub Issues
2. Create a new issue with:
   - Clear title describing the problem
   - Location of the issue (page/section)
   - Description of what's wrong or unclear
   - Suggested improvement (if applicable)

### Technical Issues

For issues with deployments or implementations:

1. Search existing issues first
2. Create a new issue with:
   - Description of the problem
   - Steps to reproduce
   - Expected vs actual behavior
   - Environment details (Azure region, services)
   - Relevant error messages or logs

## Suggesting Enhancements

We welcome suggestions for:

- New documentation topics
- Architecture patterns to document
- Additional examples or tutorials
- Tool integrations
- Best practices from your experience

Please open a GitHub Issue with the "enhancement" label describing:
- The enhancement you're proposing
- Why it would be valuable
- Any implementation ideas you have

## Documentation Style Guide

### Writing Guidelines

<AccordionGroup>
  <Accordion title="Voice and Tone">
    - Use clear, concise language
    - Write in second person ("you")
    - Be professional but approachable
    - Avoid jargon when possible, or define terms
  </Accordion>
  
  <Accordion title="Structure">
    - Use descriptive headings
    - Keep paragraphs short (3-4 sentences)
    - Use lists for sequential steps
    - Include code examples where relevant
  </Accordion>
  
  <Accordion title="Formatting">
    - Use `code formatting` for commands, parameters, and file names
    - Use **bold** for emphasis on key terms
    - Use [links](/resources/contributing) for cross-references
    - Include alt text for images
  </Accordion>
</AccordionGroup>

## Submitting Changes

### Minor Edits

For small fixes (typos, broken links, clarifications):

1. Fork the repository
2. Make your changes in a new branch
3. Submit a pull request with a clear description

### Major Contributions

For significant additions or restructuring:

1. Open an issue first to discuss the proposed changes
2. Fork the repository and create a feature branch
3. Make your changes following the style guide
4. Test locally using `mintlify dev`
5. Submit a pull request referencing the issue

## Review Process

All contributions go through review:

1. Automated checks for formatting and links
2. Maintainers review for accuracy and style
3. Feedback may be provided for requested changes
4. Approved changes are merged and deployed

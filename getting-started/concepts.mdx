# Core Concepts

Understanding these key concepts will help you navigate the AI Landing Zone documentation and implementation.

## Architecture Concepts

### Citadel Governance Hub

The **Citadel Governance Hub** is the central control plane for managing AI infrastructure across your organization. It provides:

- Centralized policy enforcement
- Unified monitoring and observability
- Cross-landing zone governance
- AI service orchestration

<Note>
  Think of Citadel as your AI infrastructure's mission control — providing visibility and control across all deployed resources.
</Note>

### Landing Zone

A **Landing Zone** is a pre-configured, secure environment for deploying AI workloads. It includes:

- Network isolation and security boundaries
- Identity and access management
- Monitoring and logging infrastructure
- AI service integration points

Each landing zone follows the principle of "secure by default" with enterprise-grade security controls.

### The 4-Layer Model

The AI Landing Zone architecture is organized into four distinct layers:

<Steps>
  <Step title="Layer 1: Foundation">
    Core Azure infrastructure including networking, identity, and security services. This layer provides the secure substrate for all AI workloads.
  </Step>
  <Step title="Layer 2: AI Services">
    Managed AI services including Azure OpenAI, AI Search, and ML workspaces. These services consume the foundation layer.
  </Step>
  <Step title="Layer 3: Governance">
    Policy enforcement, access contracts, and compliance controls. This layer ensures AI services are used responsibly.
  </Step>
  <Step title="Layer 4: Applications">
    Custom AI applications and integrations built on top of the governed AI services.
  </Step>
</Steps>

## AI Service Concepts

### API Management (APIM)

**Azure API Management** serves as the gateway for all AI service requests. It provides:

- **Rate limiting**: Control token consumption per application
- **Authentication**: Unified auth across all AI services
- **Observability**: Request/response logging and metrics
- **Transformation**: Request/response modification

### Access Contracts

An **Access Contract** defines the terms under which an application can consume AI services:

<AccordionGroup>
  <Accordion title="Rate Limits">
    Maximum requests per minute/hour/day allowed for the application.
  </Accordion>
  <Accordion title="Cost Budgets">
    Monthly or quarterly spending limits for AI service consumption.
  </Accordion>
  <Accordion title="Content Filters">
    Allowed/blocked content categories and safety thresholds.
  </Accordion>
  <Accordion title="Data Residency">
    Geographic constraints for data processing and storage.
  </Accordion>
</AccordionGroup>

### Token-Based Pricing

AI services are typically billed based on **tokens** consumed:

- **Input tokens**: Text sent to the AI model
- **Output tokens**: Text generated by the AI model
- **Pricing tiers**: Vary by model capability and region

<Tip>
  Understanding token consumption patterns is key to cost optimization. See [Operations](/operations/monitoring) for monitoring guidance.
</Tip>

## Governance Concepts

### Hub and Spoke Model

The **Hub and Spoke** network topology provides:

- **Hub**: Central network containing shared services (firewalls, DNS, monitoring)
- **Spokes**: Landing zones that connect to the hub for shared services
- **Benefits**: Centralized security, cost efficiency, simplified management

### Policy Enforcement

Azure Policy ensures compliance across all landing zones:

| Policy Category | Examples |
|-----------------|----------|
| Network Security | Require NSG on all subnets |
| Data Protection | Enforce encryption at rest |
| Access Control | Require managed identities |
| Cost Management | Enforce resource tags |

### Zero Trust Principles

The AI Landing Zone follows **Zero Trust** security principles:

1. **Verify explicitly**: Always authenticate and authorize
2. **Use least privilege**: Grant minimum necessary access
3. **Assume breach**: Design for containment and detection
